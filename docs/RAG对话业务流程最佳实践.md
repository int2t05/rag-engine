# RAG 对话业务流程最佳实践

本文档描述 RAG Engine 系统的对话流程，涵盖对话管理、检索增强生成、多知识库支持、流式响应等核心机制。

## 三、详细业务流程

### 3.1 创建对话

```
客户端                    后端                      数据库
  │                        │                         │
  │  POST /chats           │                         │
  │  {                    │                         │
  │    title: "咨询",     │                         │
  │    knowledge_base_ids: │                         │
  │      [1, 2, 3]       │                         │
  │  }                    │                         │
  │ ────────────────────▶ │                         │
  │                        │                         │
  │                        │  1. 验证知识库归属       │
  │                        │ ─────────────────────────────────────────────▶ │
  │                        │ ◀────────────────────────────────────────── │
  │                        │                         │
  │                        │  2. 创建 Chat 记录     │
  │                        │ ─────────────────────────────────────────────▶ │
  │                        │                         │
  │                        │  3. 建立多对多关联      │
  │                        │     (chat_knowledge_bases)│
  │                        │ ─────────────────────────────────────────────▶ │
  │                        │                         │
  │ ◀─────────────────── │                         │
  │  {id, title, ...}     │                         │
```

**伪代码**：
```python
def create_chat(title, knowledge_base_ids, user_id):
    # 1. 验证所有知识库存在且属于当前用户
    kbs = db.query(KnowledgeBase).filter(
        KnowledgeBase.id.in_(knowledge_base_ids),
        KnowledgeBase.user_id == user_id
    ).all()

    if len(kbs) != len(knowledge_base_ids):
        raise HTTPException(400, "部分知识库不存在或无权访问")

    # 2. 创建对话
    chat = Chat(title=title, user_id=user_id)
    chat.knowledge_bases = kbs  # 多对多关联

    db.add(chat)
    db.commit()
    return chat
```

### 3.2 发送消息（RAG 核心流程）

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        RAG 对话详细流程                                       │
└─────────────────────────────────────────────────────────────────────────────┘

  Step 1: 持久化消息
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  - 保存用户消息 (role="user")                                           │
  │  - 创建占位 AI 消息 (role="assistant", content="")                      │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
  Step 2: 加载知识库
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  - 查询对话关联的所有知识库                                              │
  │  - 为每个知识库初始化向量存储                                             │
  │  - 合并为统一检索器                                                     │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
  Step 3: 历史感知问题重写
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  使用 LLM 将对话历史中的指代词（如"它"、"上文"）转化为独立问题             │
  │                                                                           │
  │  示例：                                                                  │
  │    历史: "RAG是什么？" → "它有什么优势？"                                 │
  │    重写: "RAG的优势是什么？"                                             │
  │                                                                           │
  │  Prompt: "Given a chat history and the latest user question which      │
  │  might reference context in the chat history, formulate a standalone    │
  │  question which can be understood without the chat history..."          │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
  Step 4: 向量检索
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  - 使用重写后的问题进行向量相似度检索                                    │
  │  - 返回 top_k 个相关文档块                                               │
  │  - 每个块包含 page_content 和 metadata                                  │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
  Step 5: 构建 QA 上下文
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  将检索到的文档块拼接为上下文，要求 LLM 使用 [citation:N] 引用            │
  │                                                                           │
  │  格式:                                                                    │
  │  Context:                                                                │
  │  - [文档块1内容]                                                         │
  │  - [文档块2内容]                                                         │
  │  ...                                                                     │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
  Step 6: LLM 流式生成
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  - 调用 LLM 生成回答                                                     │
  │  - 流式返回 (SSE)                                                        │
  │  - 回答中包含 [citation:N] 引用                                          │
  └─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
  Step 7: 保存完整回答
  ┌─────────────────────────────────────────────────────────────────────────┐
  │  消息内容格式:                                                            │
  │  {Base64编码的引用上下文}__LLM_RESPONSE__{LLM生成的回答}                │
  │                                                                           │
  │  - 支持前端解析并展示引用来源                                            │
  │  - 便于调试和问题排查                                                    │
  └─────────────────────────────────────────────────────────────────────────┘
```

### 3.3 流式响应协议

```
返回格式符合 Vercel AI SDK 标准的 SSE (Server-Sent Events)：

┌─────────────────────────────────────────────────────────────────────────────┐
│                           SSE 数据格式                                        │
└─────────────────────────────────────────────────────────────────────────────┘

  事件流组成：
  1. 引用上下文（Base64编码）
     data: {"text": "eyJjb250ZXh0IjpbeyJtb2RhbCI6InBkZiJ9XX0K19LLUxfUkVTVVBMT19f"}

  2. LLM 回答片段（多次）
     data: {"text": "RAG是"}
     data: {"text": "一种结合"}
     data: {"text": "检索和"}
     data: {"text": "生成的..."}

  3. 结束标记
     data: [DONE]


┌─────────────────────────────────────────────────────────────────────────────┐
│                         前端解析流程                                          │
└─────────────────────────────────────────────────────────────────────────────┘

  1. 接收 Base64 上下文 → 解码 → 提取引用文档列表
  2. 逐步拼接 text 片段 → 显示流式回答
  3. 检测到 [citation:N] → 高亮显示引用来源
  4. 检测到 [DONE] → 结束接收
```

## 四、Prompt 设计

### 4.1 问题重写 Prompt (History-Aware)

```
System:
Given a chat history and the latest user question which might reference
context in the chat history, formulate a standalone question which can be
understood without the chat history. Do NOT answer the question, just
reformulate it if needed and otherwise return it as is.

User:
{chat_history}
{input}
```

### 4.2 QA 回答 Prompt

```
System:
You are given a user question, and please write clean, concise and accurate
answer to the question.

You will be given a set of related contexts to the question, which are
numbered sequentially starting from 1. Each context has an implicit reference
number based on its position in the array (first context is 1, second is 2,
etc.).

Please use these contexts and cite them using the format [citation:x] at the
end of each sentence where applicable.

Your answer must be correct, accurate and written by an expert using an
unbiased and professional tone.

Please limit to 1024 tokens. Do not give any information that is not related
to the question, and do not repeat.

Say 'information is missing on' followed by the related topic, if the given
context do not provide sufficient information.

If a sentence draws from multiple contexts, please list all applicable
citations, like [citation:1][citation:2].

Other than code and specific names and citations, your answer must be written
in the same language as the question.

Be concise.

Context: {context}

User:
{input}
```

### 4.3 引用格式说明

```
[citation:N] 格式：
- N 是文档块在上下文数组中的位置（从1开始）
- 第一个文档块 → [citation:1]
- 第二个文档块 → [citation:2]
- 以此类推

示例：
RAG是一种检索增强生成技术 [citation:1]。它在问答任务中表现优异 [citation:1][citation:2]。

当没有足够信息时：
information is missing on <相关主题>
```

## 六、最佳实践建议

### 6.1 对话设计建议

```
创建对话时：
✅ 明确对话主题，选择相关的知识库
✅ 一个对话聚焦一个主题效果更好
✅ 避免关联过多不相关的知识库（影响检索精度）

对话过程中：
✅ 问题尽量清晰明确
✅ 涉及上下文的问题，系统会自动重写
✅ 复杂问题可以分步提问
```

### 6.2 知识库准备建议

```
检索质量取决于知识库内容质量：

文档优化：
✅ 文档结构清晰，标题层级分明
✅ 重要信息放在段落开头
✅ 统一术语，避免同一概念多种表述
✅ 移除无关的页眉页脚水印

知识库关联：
✅ 按主题划分知识库
✅ 相关主题的知识库可同时关联到一个对话
✅ 定期更新知识库内容
```

### 6.3 引用来源展示

```
前端展示建议：

┌─────────────────────────────────────────────────────────────────────────────┐
│  用户问题: RAG有什么优势？                                                   │
│  ────────────────────────────────────────────────────────────────────────  │
│  回答:                                                                      │
│  RAG（检索增强生成）主要有以下优势：                                          │
│                                                                           │
│  1. **最新信息** [citation:1]                                               │
│     能够检索最新的文档内容，避免 LLM 知识过时的问题                            │
│                                                                           │
│  2. **可溯源** [citation:2]                                                │
│     答案直接引用自源文档，便于核实和追溯                                      │
│                                                                           │
│  3. **知识扩展** [citation:1][citation:3]                                   │
│     无需重新训练即可扩展 LLM 的知识范围                                      │
│                                                                           │
│  ────────────────────────────────────────────────────────────────────────  │
│  📄 引用来源:                                                               │
│  [1] 产品白皮书 - RAG技术介绍.pdf                                           │
│  [2] 技术文档 - 可溯源性设计.docx                                           │
│  [3] 常见问题 - FAQ.md                                                      │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 6.4 错误处理

```
常见错误及处理：

| 场景 | 返回内容 | 用户提示 |
|------|----------|----------|
| 无关联知识库 | "我没有任何知识基础..." | 请先关联知识库 |
| 检索无结果 | 基于通用知识回答 | "未找到相关内容" |
| LLM 调用失败 | 错误信息 | "生成回答失败，请重试" |
| 网络超时 | 错误信息 | "网络不稳定，请重试" |
```

## 八、性能优化建议

### 8.1 向量检索优化

```
检索参数调优：
┌─────────────────────────────────────────────────────────────────────────────┐
│  top_k 选择:                                                               │
│  - 简单问答: top_k=3~5 (太多可能引入噪声)                                  │
│  - 复杂推理: top_k=5~10 (需要更多上下文)                                    │
│  - 总结任务: top_k=10~20 (需要覆盖更多内容)                                │
└─────────────────────────────────────────────────────────────────────────────┘

相似度阈值：
- 建议设置相似度分数下限，过滤低相关结果
- Chroma 默认返回所有结果，需在后处理过滤
```

### 8.2 LLM 调用优化

```
流式响应优势：
- 首 token 时间更短，用户体验更好
- 逐步显示回答，可同步加载引用来源

Token 限制：
- 回答限制 1024 tokens
- 超出限制的回答会被截断
- 复杂问题建议分步提问
```

### 8.3 历史消息管理

```
对话历史过长的影响：
- 问题重写消耗更多 tokens
- LLM 上下文窗口压力增大
- 回答延迟增加

建议策略：
- 对话超过 20 轮后，可考虑摘要后压缩
- 或建议用户开启新对话
- 定期清理无用的对话记录
```

## 十、扩展场景

```

```

- 回答语言跟随问题语言

- 代码和专有名词不受影响

如需扩展：
- 可增加语言检测和翻译
- 支持多语言知识库混合检索

## 什么是 RAG？

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种结合"信息检索"和"文本生成"的 AI 技术模式：

```
用户提问 → 从知识库中检索相关文档片段 → 将检索结果作为上下文传给 LLM → LLM 生成带引用的回答
```

**为什么需要 RAG？**
- 纯 LLM（如 GPT-4）只知道训练数据中的知识，无法回答你私有文档中的问题
- RAG 让 LLM 能够基于你上传的文档来回答问题，且能给出引用来源
- 适用场景：企业知识库问答、文档助手、客服机器人等

**本项目实现了完整的 RAG 流程：**
1. 用户上传文档（PDF、Word、Markdown、TXT）
2. 系统将文档切分成小块（chunks），并通过 Embedding 模型转换为向量
3. 向量存储到向量数据库中（ChromaDB / Qdrant）
4. 用户提问时，系统先检索最相关的文档片段
5. 将检索到的片段作为上下文，连同问题一起发送给 LLM
6. LLM 生成带引用标记的回答，前端展示引用来源

## 项目概述

本项目是一个**全栈 RAG 应用**，包含：

| 组成部分 | 说明 |
|---------|------|
| **前端** | Next.js 14 构建的 Web 界面，提供知识库管理、文档上传、对话聊天等功能 |
| **后端** | FastAPI 构建的 API 服务，处理文档解析、向量化、检索、LLM 对话等核心逻辑 |
| **数据库** | MySQL 存储用户、知识库、文档等结构化数据 |
| **向量数据库** | ChromaDB（默认）或 Qdrant，存储文档向量用于相似度检索 |
| **对象存储** | MinIO 存储上传的原始文档文件 |
| **反向代理** | Nginx 统一转发前后端请求 |

## 核心数据流

### 流程一：文档上传与向量化

```
1. 用户在前端选择文件上传
        │
        ▼
2. 前端调用 POST /api/knowledge-base/{kb_id}/documents/upload
        │
        ▼
3. 后端将文件保存到 MinIO 对象存储（临时目录）
        │
        ▼
4. 用户点击"处理"，前端调用 POST /api/knowledge-base/{kb_id}/documents/process
        │
        ▼
5. 后端启动后台任务 process_document_background：
   ├── 5a. 从 MinIO 下载文件到本地临时目录
   ├── 5b. 使用 LangChain 文档加载器解析文件（PDF/DOCX/MD/TXT）
   ├── 5c. 使用 RecursiveCharacterTextSplitter 将文档切分成小块
   ├── 5d. 使用 Embedding 模型将文本块转换为向量
   ├── 5e. 将向量存入向量数据库（ChromaDB/Qdrant）
   ├── 5f. 在 MySQL 中创建文档记录和分块记录
   └── 5g. 将文件从临时目录移到永久目录
        │
        ▼
6. 前端轮询任务状态，显示处理进度
```

### 流程二：RAG 对话

```
1. 用户在聊天界面输入问题
        │
        ▼
2. 前端通过 Vercel AI SDK 的 useChat 发送 POST /api/chat/{chat_id}/messages
        │
        ▼
3. 后端 generate_response 函数处理：
   ├── 3a. 保存用户消息到数据库
   ├── 3b. 创建"历史感知检索器"（History-Aware Retriever）
   │        → 根据对话历史，将用户问题重写为独立问题
   ├── 3c. 使用重写后的问题从向量数据库检索相关文档片段
   ├── 3d. 构建 QA Prompt，包含检索到的上下文和引用格式要求
   ├── 3e. 将 Prompt 发送给 LLM，获取流式回答
   └── 3f. 将检索到的上下文 Base64 编码后，与回答一起流式返回
        │
        ▼
4. 前端接收流式响应：
   ├── 4a. 解码 Base64 获取引用上下文
   ├── 4b. 实时渲染 Markdown 回答
   ├── 4c. 将 [citation:N] 标记转换为可点击的引用弹窗
   └── 4d. 点击引用可查看原文来源
```

## 后端详解

### 分层架构

后端采用经典的**分层架构**，每一层有明确的职责：

```
请求 → API 层（路由） → 服务层（业务逻辑） → 模型层（数据库）
                                    ↕
                          外部服务（LLM、向量库、MinIO）
```

| 层级 | 目录 | 职责 |
|------|------|------|
| **API 层** | `app/api/` | 定义 HTTP 接口，处理请求参数校验，调用服务层 |
| **Schema 层** | `app/schemas/` | 定义请求/响应的数据格式（Pydantic 模型） |
| **服务层** | `app/services/` | 核心业务逻辑：RAG 对话、文档处理、向量操作 |
| **模型层** | `app/models/` | 数据库表定义（SQLAlchemy ORM） |
| **核心层** | `app/core/` | 通用配置、安全认证、对象存储 |
| **数据库层** | `app/db/` | 数据库连接和会话管理 |

### 关键设计模式

#### 1. 工厂模式（Factory Pattern）

项目大量使用工厂模式来支持多种后端切换：

- **LLMFactory**：根据配置创建 OpenAI 兼容 HTTP 或 Ollama 本地 LLM 实例
- **EmbeddingsFactory**：根据配置创建 OpenAI / DashScope / Ollama 的 Embedding 实例
- **VectorStoreFactory**：根据配置创建 ChromaDB / Qdrant 的向量存储实例

这样设计的好处：只需修改 `.env` 中的配置，就能切换不同的 LLM 或向量数据库，无需改动代码。

#### 2. 依赖注入（Dependency Injection）

FastAPI 的 `Depends()` 机制实现了依赖注入：

```python
@router.get("/")
def get_items(
    db: Session = Depends(get_db),           # 注入数据库会话
    current_user: User = Depends(get_current_user)  # 注入当前用户
):
    pass
```

#### 3. 流式响应（Streaming Response）

对话回答使用 Server-Sent Events (SSE) 实现流式输出：

```python
async def generate_response(...) -> AsyncGenerator[str, None]:
    async for chunk in rag_chain.astream(...):
        yield f'0:"{chunk}"\n'  # Vercel AI SDK 协议格式
```

### 数据库模型关系

```
User (用户)
├── KnowledgeBase (知识库) [一对多]
│   ├── Document (文档) [一对多]
│   │   ├── DocumentChunk (文档分块) [一对多]
│   │   └── ProcessingTask (处理任务) [一对多]
│   ├── DocumentUpload (上传记录) [一对多]
│   └── ProcessingTask (处理任务) [一对多]
├── Chat (对话) [一对多]
│   ├── Message (消息) [一对多]
│   └── KnowledgeBase [多对多，通过 chat_knowledge_bases 关联表]
└── APIKey (API 密钥) [一对多]
```

### 关键前端组件

#### Answer 组件（`components/chat/answer.tsx`）

这是最核心的前端组件之一，负责：
- 将 LLM 的 Markdown 回答渲染为 HTML
- 解析 `[citation:N]` 标记，转换为可点击的引用链接
- 点击引用时显示 Popover 弹窗，展示原文内容和来源文件

#### DocumentUploadSteps 组件（`components/knowledge-base/document-upload-steps.tsx`）

实现文档上传的三步流程：
1. **Upload（上传）**：拖拽或选择文件，上传到 MinIO
2. **Preview（预览）**：预览文档分块效果，可调整分块参数
3. **Process（处理）**：启动后台处理，轮询显示处理进度

### API 请求封装（`lib/api.ts`）

统一的 API 请求封装，提供：
- 自动附加 JWT Token 到请求头
- 统一的错误处理（401 自动跳转登录页）
- 支持 JSON、FormData、URLEncoded 等多种数据格式
- `api.get()` / `api.post()` / `api.put()` / `api.delete()` 快捷方法

## 学习建议

如果你是 RAG 新手，建议按以下顺序学习本项目：

1. **先理解 RAG 概念**：阅读本文档的"什么是 RAG？"和"核心数据流"部分
2. **从后端核心开始**：
   - `services/document_processor.py` — 理解文档如何被切分和向量化
   - `services/chat_service.py` — 理解 RAG 检索和对话生成的完整流程
   - `services/embedding/embedding_factory.py` — 理解 Embedding 的作用
   - `services/vector_store/` — 理解向量数据库的操作
3. **再看 API 层**：理解接口如何组织和调用服务层
4. **最后看前端**：理解用户交互如何与后端 API 对接

每个文件都已添加详细的中文注释，可以直接阅读代码学习！

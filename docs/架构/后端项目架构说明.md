# 后端项目架构说明

本文描述 `backend/app` 的分层与依赖方向，并与 [FastAPI](https://fastapi.tiangolo.com/)、[SQLAlchemy 2.0](https://docs.sqlalchemy.org/en/20/)、[LangChain](https://python.langchain.com/docs/) 等官方实践一致。

## 入口与生命周期

- **入口**：`app/main.py` 创建 `FastAPI` 实例，注册 `CORSMiddleware`、挂载 `api_router`，并实现 **`lifespan`**（见 [FastAPI Advanced — Events](https://fastapi.tiangolo.com/advanced/events/)）：启动时初始化 MinIO、执行 Alembic 迁移。
- **CORS**：`allow_origins=["*"]`，且 **`allow_credentials=False`**（与通配源并用 credentials 易触发浏览器/中间件问题；JWT 走 `Authorization` 头即可）。
- **根路径 `GET /`**：`Accept` 含 `text/html` 且不含 `application/json` 时 **307 重定向到 `/docs`**，便于浏览器直接打开根路径；脚本或 `curl`（不接受 HTML）仍返回 JSON 欢迎信息。
- **服务层异常**：注册 `AppServiceError` 全局处理器，经 `app/api/errors.py` 的 `http_exception_from_service` 映射为与 `HTTPException` 一致的 JSON（`detail` 字段），404/401 等会带上对应状态码与 `WWW-Authenticate`（401 时）。
- **健康检查**：`GET /api/health` 返回 `status`、`version`（`app.core.config.settings.VERSION`），供编排与探活使用。

## 分层结构

```
app/
├── main.py                 # 应用工厂、lifespan、CORS、AppServiceError 处理器、根路径与 /api/health
├── api/                    # HTTP 层：依赖注入、错误映射、v1 路由聚合
│   └── api_v1/api.py       # 将各模块 router 挂到 /api 下
├── core/                   # 横切：config、security、minio、exceptions
├── db/                     # Engine、SessionLocal
├── models/                 # SQLAlchemy ORM 模型
├── schemas/                # Pydantic v2 请求/响应模型
├── modules/                # 按业务域划分的路由与服务
│   ├── auth/
│   ├── knowledge/          # 知识库、文档、检索、处理
│   ├── chat/               # 对话、RAG 流式生成
│   ├── evaluation/       # RAGAS 等评估
│   └── llm_config/         # 用户 LLM/嵌入配置 CRUD
├── shared/                 # 跨模块复用：向量库工厂、Embedding/LLM 工厂、去重等
└── startup/                # 启动时迁移封装
```

## 依赖方向（约定）

1. **`routes`** 只负责：解析请求、调用 **service**、返回 `schemas` 或流式响应。
2. **`service`** 编排业务：访问 **`repository`**、**`models`**，调用 **`shared`** 中的工厂与向量操作。
3. **`repository`** 封装对 Session 的查询与持久化，避免在路由里写复杂 SQL。
4. **`shared`** 不依赖具体业务模块，仅依赖 `core.config`、`schemas.ai_runtime` 等与配置相关的类型。

这样便于单测替换 DB、替换向量后端实现。

## 配置

- **基础设施类**：`app/core/config.py` 使用 [pydantic-settings](https://docs.pydantic.dev/latest/concepts/pydantic_settings/) 从环境变量与项目根目录 `.env` 读取 MySQL、JWT、MinIO、Chroma 等。
- **模型运行时**：对话与嵌入使用的 API Key、Base URL、模型名等由用户在「模型配置」写入数据库，经 `load_ai_runtime_for_user` 加载为 `AiRuntimeSettings`（见 `app/shared/ai_runtime_loader.py`），与 JWT 用户上下文配合使用。

## RAG 实现要点

- 检索与生成采用 **LangChain 1.x LCEL**（链式表达式），替代已移除的旧版 `create_retrieval_chain` 等 API；详见 [LangChain 文档](https://python.langchain.com/docs/)。
- 向量存储通过 `shared/vector_store` 工厂创建，当前 MVP 以 **Chroma** 为主（[langchain-chroma](https://python.langchain.com/docs/integrations/vectorstores/chroma/)）。
- 文档解析、分块在 `modules/knowledge/document_processor.py`；长耗时同步逻辑应通过 `asyncio.to_thread` 避免阻塞事件循环（代码内已有说明）。

## 数据与迁移

- 表结构由 `app/models` 定义，版本演进由 **Alembic** 管理（[官方文档](https://alembic.sqlalchemy.org/en/latest/)），`alembic/versions` 下为迁移脚本。

## 扩展点

- **新向量库**：实现 `shared/vector_store` 中的抽象并注册工厂。
- **新 LLM 提供商**：在 `LLMFactory` / `EmbeddingsFactory` 中按 LangChain 官方集成方式扩展（保持与 `AiRuntimeSettings` 字段一致）。

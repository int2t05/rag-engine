## 模块概述

对话模块提供基于知识库的 **流式 RAG** 能力：支持多知识库关联、可配置的检索流水线（Native 向量检索为默认），以及带 `[citation:N]` 的引用与分块详情跳转。

**源码位置**: `app/modules/chat/`（核心生成逻辑：`rag_service.py`；编排：`rag/orchestrator.py`）

## API 端点

### 对话 CRUD

| 方法 | 端点 | 功能 |
|------|------|------|
| POST | `/api/chat` | 创建对话 |
| GET | `/api/chat` | 对话列表 |
| POST | `/api/chat/batch-delete` | 批量删除对话 |
| GET | `/api/chat/{chat_id}` | 对话详情 |
| DELETE | `/api/chat/{chat_id}` | 删除对话 |

### 消息与 RAG

| 方法 | 端点 | 功能 |
|------|------|------|
| POST | `/api/chat/{chat_id}/messages` | 发送消息（**SSE** 流式 RAG） |

消息内容随对话详情返回并持久化；无单独的「仅 GET 消息列表」路由。

## 发送消息请求体

Schema：`app/schemas/chat.py` → `StreamMessagesRequest`。

| 字段 | 类型 | 说明 |
|------|------|------|
| `messages` | `ChatMessageItem[]` | **完整历史**，最后一条须为 `role=user` |
| `rag_options` | `RagPipelineOptions \| null` | 可选；不传则使用后端默认（见下） |

`RagPipelineOptions` 定义见 `app/schemas/rag_pipeline.py`：

- **`top_k`**（默认 `4`，范围 1～100）：最终进入上下文与引用列表的片段数。
- **布尔开关**（默认均为 `false`）：`query_rewrite`、`multi_kb`、`hybrid`、`multi_route`、`rerank`、`parent_child`。
- **可选数值**：`rerank_top_n`、`hybrid_vector_weight`（混合检索 RRF 时向量侧权重）。

**默认行为（不传 `rag_options`）**：等价于 **Native RAG**——仅用用户原句做 **单次稠密向量检索**（第一个「已有文档且可连 Chroma」的关联知识库）、`top_k=4`，**无**查询重写、多库、混合、多路、重排、父子展开。

## RAG 流程详解

```
┌─────────────────────────────────────────────────────────────────┐
│  1. 消息落库                                                       │
│     写入用户消息 + assistant 占位行（content 稍后回填）              │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  2. 向量集合准备                                                   │
│     遍历对话关联的知识库：仅当该库下已有 Document 时创建 Chroma 客户端 │
│     集合名：kb_{id}；与 document_chunks / 向量 id 对齐               │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  3. 模块化检索（RagOrchestrator）                                  │
│     见下一节；结果写入 RagContext.retrieved_docs                   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  4. QA 提示词 + 流式生成                                           │
│     LCEL：`qa_prompt | llm | StrOutputParser()`，含 [citation:N] 要求 │
│     用户问题用原始 query；上下文为检索到的片段                      │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  5. SSE 下发                                                       │
│     首包（或首段）：Base64(JSON{context:[{page_content,metadata}]}) │
│     + 分隔符 __LLM_RESPONSE__ + 后续增量文本块                      │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  6. 写回 assistant 消息的 content（含 Base64 引用前缀 + 正文）     │
└─────────────────────────────────────────────────────────────────┘
```

## 模块化检索流水线

执行类：`app/modules/chat/rag/orchestrator.py` → `RagOrchestrator.run_retrieval_pipeline`。

顺序与 [LangChain RAG 教程](https://python.langchain.com/docs/tutorials/rag/) 中「检索前 / 检索 / 检索后」划分一致：

| 顺序 | 步骤 | 开关字段 | 说明 |
|------|------|----------|------|
| 1 | 查询重写 | `query_rewrite` | 历史感知，将多轮指代改写为独立检索句（LCEL + LLM） |
| 2 | 多路召回 | `multi_route` | LLM 生成多条子查询，检索后合并去重 |
| 3 | 稠密 + 可选混合 | `hybrid`、`multi_kb` | `retrieve_for_context`（`rag/retrieval_core.py`）：向量检索；`hybrid` 时叠加 MySQL 全库 BM25 + RRF |
| 4 | 父子块展开 | `parent_child` | 若元数据含 `parent_chunk_id`，用父块全文替换子块正文再进上下文 |
| 5 | 重排与截断 | `rerank` | FlashRank 重排（可选）；最终截断至 `top_k` |

**依赖**：混合检索依赖 `rank-bm25`；重排依赖 `flashrank`（见根目录 `backend/requirements.txt`）。

**父子分块入库**：优先由知识库 **`parent_child_chunking`**（前端新建/编辑知识库）或环境变量 **`RAG_PARENT_CHILD_INGEST`** 启用；对话侧需同时勾选 `parent_child` 才会在检索后展开父块。

## SSE 流式响应

### 响应格式

```text
data: {"text": "<base64_json_context>__LLM_RESPONSE__<可选首段正文>"}
data: {"text": "<后续正文块>"}
...
data: [DONE]
```

Base64 解码后为 UTF-8 JSON：`{"context":[{"page_content":"...","metadata":{...}}, ...]}`。`metadata` 中含 `kb_id`、`chunk_id`、`document_id`、`source` 等，供前端「引用详情」跳转（`GET /api/knowledge-base/{kb_id}/chunks/{chunk_id}`）。

### 前端解析要点

1. 按行读取 `data:`，解析 JSON 取 `text`。
2. 若含 `__LLM_RESPONSE__`，前半为引用 Base64，后半为模型正文片段。
3. 引用列表与历史消息解析逻辑见前端 `lib/rag-context.ts`、`lib/chat-stream.ts`、`hooks/useChatSession.ts`。

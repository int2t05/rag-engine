# RAG 评估业务流程最佳实践

基于 RAGAS 评估框架的 RAG 系统评估指南

---

## 一、评估指标体系

本项目采用 RAGAS 评估框架，包含以下核心指标：

| 指标 | 说明 | 评估对象 | 理想值 |
|------|------|----------|--------|
| **Context Relevance** | 检索到的上下文与问题的相关程度 | 检索模块 | ≥ 0.6 |
| **Faithfulness** | 生成的答案是否忠实于检索到的上下文 | 生成模块 | ≥ 0.6 |
| **Answer Relevancy** | 生成的答案与问题的相关程度 | 生成模块 | ≥ 0.6 |
| Context Precision | 相关上下文在检索结果中的排名质量 | 检索模块 | ≥ 0.5 |
| Context Recall | 检索到的上下文覆盖参考答案的程度 | 检索模块 | ≥ 0.5 |

> **通过标准**：三个核心指标（Context Relevance、Faithfulness、Answer Relevancy）均达到 0.6 及以上视为通过。

### 2.2 详细步骤说明

#### Step 1: 创建评估任务

```
输入: 任务名称、知识库ID、配置参数
输出: 评估任务ID
```

**伪代码**：
```
1. 验证知识库归属（确保用户有权限）
2. 创建 EvaluationTask 记录
   - name: 任务名称
   - knowledge_base_id: 关联知识库
   - top_k: 检索数量（默认5）
   - evaluation_type: full/retrieval/generation
   - status: "pending"
3. 返回任务ID
```

#### Step 2: 添加测试用例

```
输入: 任务ID、测试用例列表（query + reference）
输出: 测试用例ID列表
```

**伪代码**：
```
1. 验证任务归属和状态
2. 对每个测试用例:
   - query: 用户可能提出的问题
   - reference: 参考答案（用于 Context Recall 评估）
3. 批量插入 EvaluationTestCase 记录
```

**测试用例设计原则**：
- 覆盖性：涵盖知识库的主要 topic
- 多样性：包含简单事实问答、推理题、总结题等
- 挑战性：包含边界情况和歧义性问题
- 数量建议：每个知识库至少 20-30 个测试用例

#### Step 3: 触发评估执行

```
输入: 任务ID
输出: 执行确认
```

**伪代码**：
```
1. 验证任务状态（非 running）
2. 更新状态为 "running"
3. 提交后台任务
4. 返回 "评估已启动"
```

#### Step 4: 后台异步评估

```
输入: 任务ID
输出: 评估结果
```

**伪代码**：
```
对于每个测试用例:
    1. 检索阶段
       - 使用 RetrievalForEvaluation 对 query 进行检索
       - 获取 top_k 个相关文档片段
       - 保存 retrieved_contexts

    2. 生成阶段
       - 使用 AnswerGenerator 基于上下文生成答案
       - 遵循"仅依据上下文回答"原则
       - 保存 generated_answer

    3. 评估阶段
       - 构建 EvaluationSample (question, answer, contexts, ground_truth)
       - 调用 RagasEvaluator 进行多指标评估
       - 获取各指标得分

    4. 结果保存
       - 创建 EvaluationResult 记录
       - 保存各项指标得分
       - 判定 passed 状态

汇总统计:
   - 计算各项指标平均值
   - 计算通过率
   - 更新任务 summary
   - 更新任务状态为 "completed"
```

#### Step 5: 获取评估报告

```
输入: 任务ID
输出: 评估结果列表 + 汇总统计
```

## 三、评估类型配置

| 类型 | 评估内容 | 适用场景 |
|------|----------|----------|
| `full` | 检索 + 生成 + 全指标评分 | 完整评估 |
| `retrieval` | 仅检索 + 检索指标 | 专注优化检索 |
| `generation` | 仅生成 + 生成指标 | 专注优化生成 |

## 四、RAGAS 模型与环境配置

评估链路中的**生成答案**与 **RAGAS 指标打分**共用聊天模型；**向量检索**与部分 RAGAS 指标（如需 embedding 的相似度类）共用 Embedding。与业务保持一致即可，避免「入库一套模型、评估另一套模型」导致指标失真。

| 用途 | 配置入口 | 说明 |
|------|----------|------|
| 对话 / 生成答案 / RAGAS 的 LLM 判分 | `CHAT_PROVIDER` + `OPENAI_*` 或 `OLLAMA_*` | 任意 OpenAI 兼容服务：改 `OPENAI_API_BASE` + `OPENAI_API_KEY` + `OPENAI_MODEL`；本地用 `ollama` + `OLLAMA_*` |
| 知识库检索与 RAGAS 所需向量 | `EMBEDDINGS_PROVIDER`：`openai` 或 `ollama` | 须与入库一致；嵌入与对话不同 URL 时用 `OPENAI_EMBEDDINGS_API_BASE`（可选 `OPENAI_EMBEDDINGS_API_KEY`）或 `OLLAMA_EMBEDDINGS_API_BASE`，未设则回退 `OPENAI_API_*` / `OLLAMA_API_BASE` |

**建议**：评分任务使用较低温度；本项目中评估路径对 LLM 工厂传入 `temperature=0`，与《注意事项》中「评估一致性」一致。完整变量名见项目根目录 `.env.example` 与 `backend/app/core/config.py`。

## 五、最佳实践建议

### 5.1 测试用例设计

#### 好的测试用例示例
```
✅ query: "什么是 RAG？"
   reference: "RAG(Retrieval-Augmented Generation)是一种结合检索和生成的AI技术..."

✅ query: "知识库最近更新了哪些内容？"
   reference: "知识库于2024年1月更新了文档处理模块..."

✅ query: "如果用户问题超出知识库范围，应该如何处理？"
   reference: "应明确告知用户当前知识库无法回答该问题..."
```

#### 需避免的测试用例
```
❌ query: "介绍你自己"
   → 与知识库内容无关

❌ query: "总结所有文档"
   → 过于宽泛，无法验证具体能力

❌ query: "那关于 XX 呢？"（指代不清）
   → 缺乏上下文无法评估
```

### 5.2 top_k 参数选择

| 知识库规模 | 建议 top_k | 说明 |
|------------|------------|------|
| 小型（<100篇） | 3-5 | 文档少，过多检索可能导致噪声 |
| 中型（100-1000篇） | 5-10 | 平衡相关性和多样性 |
| 大型（>1000篇） | 10-15 | 需要足够上下文覆盖复杂问题 |

### 5.3 评估频率建议

| 场景 | 建议频率 | 说明 |
|------|----------|------|
| 日常监控 | 每周一次 | 监控系统稳定性 |
| 上线前验收 | 每次重大更新前 | 确保质量达标 |
| 调优迭代 | 每次参数调整后 | 验证优化效果 |

### 5.4 评估结果分析流程

```
                    ┌─────────────────┐
                    │   评估报告      │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
      ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
      │ Context低     │ │ Faithfulness低│ │ Answer低     │
      │ (检索问题)     │ │ (生成幻觉)    │ │ (Prompt问题) │
      └──────┬───────┘ └──────┬───────┘ └──────┬───────┘
             │                 │                 │
             ▼                 ▼                 ▼
    ┌────────────────┐ ┌────────────────┐ ┌────────────────┐
    │ 1. 检查Embedding│ │ 1. 增加上下文约束│ │ 1. 优化Prompt  │
    │ 2. 优化分块策略 │ │ 2. 添加忠实度指令│ │ 2. 调整模型    │
    │ 3. 调整top_k    │ │ 3. 减少生成长度 │ │ 3. 增加示例    │
    └────────────────┘ └────────────────┘ └────────────────┘
```

### 5.5 评估失败问题定位

| 指标异常 | 可能原因 | 排查方向 |
|----------|----------|----------|
| Context Relevance 低 | - Embedding 模型不匹配<br>- 分块策略不当<br>- 知识库内容缺失 | 抽查检索结果质量 |
| Faithfulness 低 | - 检索上下文不足<br>- Prompt 约束不够<br>- 模型幻觉倾向 | 检查答案与上下文一致性 |
| Answer Relevancy 低 | - 问题与答案不匹配<br>- 上下文关键信息被遗漏<br>- 答案组织结构差 | 人工审查问答对 |

## 八、注意事项

1. **评估成本**：RAGAS 评估涉及多次 LLM 调用，需控制测试用例数量
2. **参考答案质量**：参考答案直接影响 Context Recall，建议人工审核
3. **评估一致性**：相同条件下多次评估结果应保持稳定（建议 temperature=0）
4. **异步执行**：大型评估任务耗时长，必须后台执行避免超时
5. **结果持久化**：评估结果应持久化存储，便于历史对比


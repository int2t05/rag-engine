# 用户认证业务流程最佳实践

本文档描述 RAG Engine 系统的用户认证体系，包括 JWT Token 认证和 API Key 认证两种方式。

## 二、认证方式

| 方式 | 适用场景 | Token 类型 | 有效期 |
|------|----------|-----------|--------|
| **JWT Token** | 前端用户登录 | Access Token | 可配置（默认7天） |
| **API Key** | 外部系统/程序调用 | 静态密钥 | 长期有效，直到删除 |

## 三、用户认证流程

### 3.1 用户注册

```
┌─────────────────────────────────────────┐
│            用户注册流程                   │
└─────────────────────────────────────────┘

  前端                      后端                      数据库
   │                         │                          │
   │  POST /auth/register    │                          │
   │  {email, username,      │                          │
   │   password}              │                          │
   │ ───────────────────────▶ │                          │
   │                         │                          │
   │                         │  1. 检查 email 唯一性     │
   │                         │ ─────────────────────────▶│
   │                         │ ◀──────────────────────── │
   │                         │    (如已存在 → 400)       │
   │                         │                          │
   │                         │  2. 检查 username 唯一性   │
   │                         │ ─────────────────────────▶│
   │                         │ ◀──────────────────────── │
   │                         │    (如已存在 → 400)       │
   │                         │                          │
   │                         │  3. bcrypt 哈希密码        │
   │                         │                          │
   │                         │  4. 创建用户记录           │
   │                         │ ─────────────────────────▶│
   │                         │                          │
   │ ◀────────────────────── │                          │
   │  201 Created            │                          │
   │  {user_id, email,       │                          │
   │   username}             │                          │
```

**伪代码**：
```python
def register(email, username, password):
    # 1. 验证邮箱唯一性
    if db.users.find(email=email):
        raise HTTPException(400, "邮箱已被注册")

    # 2. 验证用户名唯一性
    if db.users.find(username=username):
        raise HTTPException(400, "用户名已被占用")

    # 3. 密码安全检查（建议补充）
    if len(password) < 8:
        raise HTTPException(400, "密码长度至少8位")

    # 4. bcrypt 哈希密码（自动加盐）
    hashed = bcrypt.hash(password)

    # 5. 创建用户
    user = User(
        email=email,
        username=username,
        hashed_password=hashed,
        is_active=True
    )
    db.add(user)
    db.commit()
    return user
```

### 3.2 用户登录 (获取 Token)

```
┌─────────────────────────────────────────┐
│            用户登录流程                   │
└─────────────────────────────────────────┘

  前端                      后端                      数据库
   │                         │                          │
   │  POST /auth/token       │                          │
   │  (form-data:            │                          │
   │   username, password)   │                          │
   │ ───────────────────────▶ │                          │
   │                         │                          │
   │                         │  1. 查找用户              │
   │                         │ ─────────────────────────▶│
   │                         │ ◀──────────────────────── │
   │                         │                          │
   │                         │  2. 验证明文密码          │
   │                         │    bcrypt.checkpw()      │
   │                         │                          │
   │                         │  3. 检查用户激活状态       │
   │                         │                          │
   │                         │  4. 生成 JWT Token        │
   │                         │    {sub: username,       │
   │                         │     exp: +7days}         │
   │                         │                          │
   │ ◀────────────────────── │                          │
   │  200 OK                 │                          │
   │  {access_token,        │                          │
   │   token_type: "bearer"}│                          │
```

**伪代码**：
```python
def login(username, password):
    # 1. 查找用户
    user = db.users.find(username=username)
    if not user:
        raise HTTPException(401, "用户名或密码错误")

    # 2. 验证密码
    if not bcrypt.checkpw(password, user.hashed_password):
        raise HTTPException(401, "用户名或密码错误")

    # 3. 检查激活状态
    if not user.is_active:
        raise HTTPException(401, "用户未激活")

    # 4. 生成 JWT Token
    token_data = {
        "sub": user.username,
        "exp": datetime.utcnow() + timedelta(minutes=10080)  # 7天
    }
    access_token = jwt.encode(token_data, SECRET_KEY, algorithm="HS256")

    return {"access_token": access_token, "token_type": "bearer"}
```

### 3.3 Token 验证 (受保护接口访问)

```
┌─────────────────────────────────────────┐
│         受保护接口访问流程                 │
└─────────────────────────────────────────┘

  前端                      后端
   │                         │
   │  GET /protected         │
   │  Authorization:         │
   │  Bearer <access_token> │
   │ ───────────────────────▶│
   │                         │
   │                         │  1. 从 Header 提取 Token
   │                         │  2. JWT 解码验证签名
   │                         │  3. 检查过期时间
   │                         │  4. 获取 username
   │                         │  5. 查询数据库获取用户
   │                         │
   │                         │  6. 注入 current_user
   │                         │     到路由函数
   │                         │
   │ ◀────────────────────── │
   │  200 OK + 资源数据      │
```

**依赖注入方式**：
```python
# 在需要认证的路由中使用
@router.get("/knowledge-bases")
async def list_kbs(current_user: User = Depends(get_current_user)):
    # current_user 自动注入，无需手动验证
    return db.query(KnowledgeBase).filter(user_id=current_user.id).all()
```

## 四、API Key 认证流程

### 4.1 创建 API Key

```
用户请求                后端                      数据库
    │                    │                          │
    │ POST /api-keys     │                          │
    │ {name: "我的Key"}  │                          │
    │ ────────────────▶ │                          │
    │                    │                          │
    │                    │  1. 生成随机 Key         │
    │                    │    "sk-" + 64位十六进制  │
    │                    │                          │
    │                    │  2. 存储到 api_keys 表   │
    │                    │ ───────────────────────▶│
    │                    │                          │
    │ ◀─────────────── │                          │
    │ 201: {key: "sk-...", │                       │
    │     name: "我的Key"}│                          │
    │
    │ ⚠️ 立即返回给用户   │
    │   (只显示一次!)    │
```

**伪代码**：
```python
def create_api_key(user_id, name):
    # 生成安全的随机 Key
    api_key = f"sk-{secrets.token_hex(32)}"

    # 存储记录
    key_record = APIKey(
        key=api_key,
        name=name,
        user_id=user_id,
        is_active=True,
        created_at=datetime.utcnow()
    )
    db.add(key_record)
    db.commit()

    # ⚠️ 只在此刻返回明文 Key，后续无法找回
    return {"key": api_key, "name": name}
```

```
外部系统                后端                      数据库
    │                    │                          │
    │  GET /api/...     │                          │
    │  X-API-Key:        │                          │
    │  sk-xxxxx...       │                          │
    │ ────────────────▶ │                          │
    │                    │                          │
    │                    │  1. 提取 X-API-Key      │
    │                    │  2. 查找对应 Key 记录    │
    │                    │ ───────────────────────▶│
    │                    │ ◀───────────────────────│
    │                    │                          │
    │                    │  3. 检查激活状态          │
    │                    │                          │
    │                    │  4. 更新最后使用时间       │
    │                    │ ───────────────────────▶│
    │                    │                          │
    │                    │  5. 返回关联用户          │
    │                    │                          │
    │ ◀─────────────── │                          │
    │  200 OK           │                          │
```

**依赖注入方式**：
```python
# 在 API Key 认证的路由中使用
@router.get("/openapi/chat")
async def chat(
    query: str,
    current_user: User = Depends(get_api_key_user)
):
    return {"user": current_user.username, "query": query}
```

### 4.3 API Key 安全建议

| 安全措施 | 说明 |
|----------|------|
| Key 一次性显示 | 创建时只返回一次，之后不再明文展示 |
| HTTPS 传输 | 生产环境必须使用 HTTPS |
| 定期轮换 | 建议定期创建新 Key 并废弃旧 Key |
| 最小权限 | 不同系统使用不同 Key，便于权限控制 |
| 使用记录审计 | 记录最后使用时间，便于异常检测 |

## 五、安全配置说明

### 5.1 关键配置项

| 配置项 | 默认值 | 说明 | 安全建议 |
|--------|--------|------|----------|
| `SECRET_KEY` | change-me | JWT 签名密钥 | 必须修改为随机长字符串 |
| `ALGORITHM` | HS256 | JWT 算法 | 不建议更改 |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | 10080 (7天) | Token 有效期 | 根据业务调整 |

### 5.2 密码安全

```
bcrypt 哈希特点：
┌─────────────────────────────────────────────────────┐
│  1. 自动加盐 - 同一密码每次哈希结果不同             │
│  2. 计算成本高 - 抗暴力破解                        │
│  3. 不可逆 - 无法从哈希反推明文                    │
└─────────────────────────────────────────────────────┘

密码验证流程：
明文密码 ──▶ bcrypt.checkpw ──▶ 对比哈希值
```

### 5.3 JWT Token 结构

```
┌─────────────────────────────────────────────────────┐
│                   JWT Token 组成                     │
├─────────────┬───────────────────────────────────────┤
│   Header    │  {"alg": "HS256", "typ": "JWT"}       │
├─────────────┼───────────────────────────────────────┤
│   Payload   │  {"sub": "username",                  │
│             │   "exp": 1234567890}                  │
├─────────────┼───────────────────────────────────────┤
│   Signature │  HMAC-SHA256(Header.Payload, SECRET) │
└─────────────┴───────────────────────────────────────┘
```

## 六、常见安全问题与防护

### 6.1 暴力破解防护

| 威胁 | 防护措施 |
|------|----------|
| 密码暴力猜解 | 建议增加登录失败次数限制（本项目可扩展） |
| Token 伪造 | JWT 签名验证，密钥足够复杂 |
| API Key 枚举 | 使用足够长的随机密钥 (sk- + 64位十六进制) |

### 6.2 Token 安全使用

```
✅ 正确做法：
   - Token 存储在安全位置（HttpOnly Cookie 或内存）
   - 使用 HTTPS 传输
   - 敏感操作定期要求重新登录

❌ 避免做法：
   - Token 存储在 localStorage（XSS 风险）
   - Token 在 URL 中传递（会被日志记录）
   - 永不过期的 Token
```

### 6.3 用户权限控制

```
用户类型：
┌─────────────────────────────────────────────────────┐
│                   User                              │
│  ├── is_active: False → 账户禁用，所有认证失败      │
│  └── is_superuser: True → 管理员权限（谨慎分配）    │
└─────────────────────────────────────────────────────┘

权限检查示例：
def get_current_user(...):
    user = db.query(User).filter(...).first()
    if not user.is_active:
        raise HTTPException(401, "用户未激活")
    return user
```

## 七、最佳实践总结

### 7.1 开发阶段

| 检查项 | 说明 |
|--------|------|
| 修改 SECRET_KEY | 生产部署前必须修改为随机值 |
| 启用 HTTPS | 生产环境必须使用 HTTPS |
| 密码强度策略 | 建议要求至少8位，包含字母和数字 |

### 7.3 错误处理

| 状态码 | 场景 | 处理建议 |
|--------|------|----------|
| 400 | 参数错误/邮箱已注册 | 返回具体错误信息 |
| 401 | 认证失败/Token无效 | 提示重新登录 |
| 403 | 权限不足 | 提示权限不足 |

## 八、扩展建议

### 8.1 可增加的增强功能

| 功能 | 说明 |
|------|------|
| 登录失败锁定 | 连续失败5次后锁定账户15分钟 |
| Refresh Token | 实现 Token 自动续期 |
| 邮箱验证 | 注册后发送验证邮件 |
| 密码重置 | 邮件链接重置密码 |
| 双因素认证 | TOTP 或短信验证码 |
| 登录日志 | 记录每次登录的 IP 和时间 |
| API Key 权限细分 | 按功能模块划分 Key 权限 |

### 8.2 审计日志建议

建议记录以下认证事件：
- 用户注册
- 登录成功/失败
- Token 刷新
- API Key 创建/删除/使用
- 权限变更

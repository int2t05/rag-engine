# 06 · 风险合规与综合开放题

## 1. 知识库里可能有敏感信息，业务上要考虑什么？

**答题要点**

| 层面 | 考虑点 |
|------|--------|
| **访问控制** | 谁建库、谁可见、是否按部门隔离（产品级常需扩展） |
| **传输与存储** | HTTPS、密钥轮换、MinIO 访问策略 |
| **日志** | 对话与检索日志是否含 PII，留存周期 |
| **导出** | 防止批量爬取知识库 |
| **合规** | 跨境模型调用、行业监管（金融/医疗） |

**与本项目对应**

- 当前示例以功能闭环为主；面试中可明确「生产需加固权限与审计」，并与根 [README.md](../../README.md) 中「生产与安全」待办呼应。

---

## 2. 用户发现「引用和内容对不上」，可能原因有哪些？怎么对外沟通？

**答题要点**

- **可能原因**：索引未更新、检索错段、模型概括偏离原文、前端展示 bug、同一文档多版本混淆。
- **对外**：先**道歉与核实**，提供原文链接；对内走**工单**与**复现**。
- **长期**：版本号、处理任务成功后再提示「可对话」、评估集加入此类 bad case。

---

## 3. 「回答不准」时，不从代码出发，你的排查框架是什么？

**答题要点（业务顺序）**

1. **知识是否覆盖**：库里是否真有答案？是否过期？
2. **是否选对知识库**：多库场景是否选错范围？
3. **问题是否清晰**：是否需要用户补充背景？
4. **检索策略是否过弱**：是否需要混合/重排/改写（与产品开关相关）？
5. **生成是否过度发挥**：是否需更保守的提示词或强制引用原文？
6. **模型能力**：小模型是否不足以综合多段？

**与本项目对应**

- 可点名 `RagPipelineOptions` 各开关与 `top_k`、去重逻辑，体现你懂**产品+技术**联动。
- `rag_dedupe.py` 去重失败可能导致引用冗余；`parent_child` 展开异常可能导致上下文截断。

## 4. 若要做成商业化产品，你会讲哪三个差异化卖点？

**答题要点（结合本仓库）**

1. **可配置检索管线**：从简单向量到混合、多路、重排、父子分块，可随成熟度渐进打开。
2. **模型与数据路径清晰**：MySQL + Chroma + MinIO 分层，用户级模型配置，利于私有化与对接多供应商。
3. **质量闭环**：对话流式 + 引用溯源 + 可选 RAGAS 评估，形成「上线—测—改」迭代。

（可按面试官行业改写成客服/内部知识/销售助手版本。）

## 5. 开放题：「如果只有一周做演示，你砍什么留什么？」

**答题要点**

- **留**：一条完整路径（登录 → 配模型 → 建库 → 上传成功 → 对话有引用）。
- **砍或降级**：评估大屏、复杂策略默认关、非核心页面美化。
- **原则**：**可讲清故事**比「功能清单长」更重要。

---

## 6. 开放题：「和买微软 Copilot / 某 SaaS 比，你们自研的价值？」

**答题要点**

- **数据驻留与合规**：自建可控制区域与审计。
- **深度定制**：行业流程、与内部系统集成（工单、CRM）。
- **成本结构**：长期大量 Token 时自研优化空间大。
- **诚实**：通用 SaaS 迭代快、省团队；自研适合强合规与深集成。

---

## 7. 开放题：「如何收集用户反馈改进 RAG？」

**答题要点**

- **显式**：点踩、选「未解决原因」（没搜到/答非所问/引用错）。
- **隐式**：重复提问、会话长度、引用点击率。
- **闭环**：每周把 bad case 变成**评估集**或 golden set，再改检索或提示词。

## 小结（背诵用）

- 合规题：**访问、日志、密钥、留存**四维至少提到。
- 排障题：**知识→范围→问题→检索→生成→模型**顺序。
- 开放题：**先讲商业取舍**，再带一句本项目能力锚点，显得落地。
- 技术亮点要能用**一句话 + 一个组件名 + 一个数字**的方式背出来。

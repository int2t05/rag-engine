# 04 · 对话检索与体验

## 1. 为什么对话回答要用流式（SSE）？业务价值是什么？

**答题要点**

- **用户侧**：**首字更快出现**，体感延迟低；长回答可边读边判断是否停止。
- **产品侧**：可中途展示「正在生成」、后续可扩展**中途取消**（若前端支持）。
- **对比一次性返回**：大包体等待更焦虑，流式更符合聊天心智。

**与本项目对应**

- `POST /api/chat/{chat_id}/messages` 使用 SSE 流式 RAG（见 [../架构/API路由.md](../架构/API路由.md)）。

---

## 2. 「引用」在业务上解决什么问题？没有引用会怎样？

**答题要点**

- **解决**：**溯源**（哪段话）、**核对**（点开看原文）、**合规**（对外答复需出处）。
- **没有引用**：用户无法验证，容易沦为「又一个聊天机器人」，**难以进入严肃场景**（客服政策、法务、医疗免责声明等）。

**与本项目对应**

- 前端展示引用并可跳转分块/文档详情页（与知识库 `chunks`、`documents` 路由对应）。

## 3. 如何把「混合检索、重排、多路召回、查询重写」用业务语言讲清楚？

**答题要点（类比）**

| 能力 | 业务类比 |
|------|-----------|
| **查询重写** | 把「它」「上面说的」补全成完整问题再去搜 |
| **混合检索** | 既按**意思相近**找，又按**关键词命中**找，再合并 |
| **多路召回** | 多问几个子问题分头搜索，最后汇总，减少「只搜到一个偏门角度」 |
| **重排** | 粗筛 50 条后再**精读排序**，把最相关的放前面 |
| **父子分块** | 先锁定句子，再展开整段政策原文给模型读 |

**延伸：管线开关伪代码对照**

```python
# RagPipelineOptions 开关定义（app/schemas/rag_pipeline.py）
class RagPipelineOptions(BaseModel):
    top_k: int = 4                        # 最终参与生成与引用的片段数
    query_rewrite: bool = False           # 历史感知查询重写
    multi_kb: bool = False               # 多知识库分别检索合并去重
    hybrid: bool = False                  # 向量 + BM25 混合检索，RRF 融合
    multi_route: bool = False            # 多路召回：LLM 生成子查询并行检索
    rerank: bool = False                  # FlashRank 交叉编码器重排
    parent_child: bool = False            # 展开 parent_chunk_id 为父块全文
    rerank_top_n: int | None = None       # 重排前候选数，默认 max(top_k*4, 16)
    hybrid_vector_weight: float = 0.5     # RRF 中向量侧权重
```

**与本项目对应**

- 字段：`hybrid`、`rerank`、`multi_route`、`query_rewrite`、`parent_child` 等（`RagPipelineOptions`）。

## 4. 检索结果「去重」为什么要做？对用户可见的影响？

**答题要点**

- **原因**：同一分片可能被多条向量或多次检索路径命中，**重复片段**会浪费上下文窗口、让引用列表又臭又长。
- **用户可见影响**：引用更干净、回答更聚焦；减少「三条引用其实是一段话」的糟糕体验。

**延伸：去重实现伪代码**

```python
# app/shared/rag_dedupe.py
def dedupe_retrieved_documents(docs: List[Any]) -> List[Any]:
    seen: set = set()
    out: []
    for doc in docs:
        key = _doc_dedupe_key(doc)  # 生成去重键
        if key in seen:
            continue
        seen.add(key)
        out.append(doc)
    return out

def _doc_dedupe_key(doc):
    md = dict(doc.metadata or {})
    # 父子块展开后：多子块命中同一父块，按 parent_chunk_id 合并
    if md.get("expanded_from_child") and md.get("parent_chunk_id"):
        return ("parent", str(md["parent_chunk_id"]))
    # 普通块按 chunk_id 去重
    if md.get("chunk_id"):
        return ("cid", str(md["chunk_id"]))
    # 回退：按来源+内容
    return ("fb", md.get("source"), doc.page_content)
```

**与本项目对应**

- `app/shared/rag_dedupe.py`：按 `chunk_id` 或父子展开后的父块键合并。

## 5. `top_k` 在业务上怎么解释？调大调小分别会怎样？

**答题要点**

- **含义**：最终参与生成与引用的**片段条数上限**（产品话术：「最多参考几段材料」）。
- **调大**：覆盖面广，但可能引入噪声、延迟与成本上升。
- **调小**：更聚焦，但可能漏掉关键段落。
- **加分**：可与重排、混合检索配合——先宽后窄，而不是一味增大 k。

## 6. 多轮对话里，用户说「详细说说第二点」，系统可能出什么问题？

**答题要点**

- **检索侧**：若只用当前一句，可能**丢上下文**，检索词不完整 → 需要**查询重写**或把历史拼进检索 query（策略问题）。
- **生成侧**：模型可能**重复**已说内容，需要摘要或指代消解（产品可提示用户「请指明是哪一条政策」）。

**本项目应对**：`query_rewrite=True` 时启用历史感知查询重写，将多轮历史拼完整后再检索。

---

## 7. 追问：「空检索」时产品应该怎么做，而不是让模型瞎编？

**答题要点**

- **明确告知**：知识库中未找到相关内容。
- **引导**：建议换关键词、检查是否选对知识库、联系管理员补充文档。
- **底线**：禁止模型在**无依据**时编造条款；可配置保守话术（实现层可用「无检索结果」分支，属扩展话题）。

---

## 小结（背诵用）

- **SSE** = 体验与可控；**引用** = 信任与合规。
- 高级检索用**类比**讲，避免堆砌名词。
- **去重**与 **top_k** 都能用「用户看到几条引用、花多少上下文」来解释。

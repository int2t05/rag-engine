## 一、项目概览与定位

### Q1：用一句话介绍你做的这个 RAG Engine 项目

**答：**
> 这是一个基于 FastAPI + LangChain 构建的企业级 RAG（检索增强生成）引擎，提供从知识库管理、智能对话到效果评估的完整闭环。简单说就是：让大模型"读"企业自己的文档，然后回答用户问题，并且告诉你答得好不好。

---

### Q2：这个项目和市面上现有的 RAG 产品比，核心差异是什么？

**答（可任选 2-3 点展开）：**

| 维度 | 通用方案 | 本项目 |
|------|---------|--------|
| **检索管线** | 简单向量检索 | 混合检索 + RRF 融合 + FlashRank 重排 + 父子分块，可按场景开关 |
| **评估体系** | 无或简陋 | RAGAS 五维全链路自动化评估，可批量跑、可对比 |
| **模型配置** | 写死在代码或环境变量 | 用户级配置入库，ContextVar 请求级隔离，0 代码切换 |
| **知识库生态** | 单库或无版本管理 | 多知识库隔离 + 增量更新 + hash 去重 + 引用溯源 |

---

### Q3：你觉得这个项目最有技术挑战的部分是哪块？

**答（推荐选检索管线或评估）：**

**推荐方向一：Advanced RAG 检索流水线**
> 检索是 RAG 的核心，效果不好后面全白搭。我们实现了一套"组合拳"——混合检索（向量 + BM25 稀疏）用 RRF 融合，然后 FlashRank 交叉编码器重排，再配合父子分块策略。实测问答准确率比纯大模型提升 45% 以上。过程中踩过不少坑，比如 RRF 权重怎么设、重排候选数怎么配、父子块展开时机等。

**推荐方向二：RAGAS 评估体系**
> 评估的难点是怎么做到"可对比、可复现"。我们接入了 RAGAS 五维指标，评估任务和对话共用同一套检索管线配置，保证对比公平性。实现过程中最麻烦的是 JSON 解析稳定性——LLM 输出格式一旦变化整个评估就挂了，所以用了 Strict JSON 提示词工程来解决。

## 二、架构与设计

### Q4：画一下你们的系统架构

**答（边画边说）：**

```
用户请求
   │
   ▼
FastAPI（Routes层：认证/知识库/对话/评估/模型配置）
   │
   ▼
Service层（业务逻辑编排）
   │
   ├──► SQLAlchemy + MySQL（用户、配置、知识库、对话、评估）
   ├──► MinIO（文档对象存储）
   └──► ChromaDB（向量数据库）
            ▲
            │
   LLMFactory / EmbeddingsFactory（OpenAI / Ollama）
```

**补充说明：**
- 请求进来先过 JWT 认证，然后根据路由分发
- 知识库相关写 MySQL + MinIO + Chroma
- 对话时从数据库加载用户配置的模型参数，通过 ContextVar 注入到当前请求上下文
- 所有 AI 调用（LLM/Embedding）都走工厂类，屏蔽底层差异

### Q5：为什么选择 ContextVar 而不是直接传参？

**答：**
> 这是个很好的问题。ContextVar 是 Python 3.7+ 引入的上下文变量，最适合的场景就是**异步请求级**的数据传递。

**痛点场景：**
- 用户 A 用 GPT-4，用户 B 用 Ollama，两个请求同时进来
- 如果用普通全局变量，两人都拿到同一个模型——那就乱套了
- 如果每个函数都加个 `settings` 参数，调用链会变得又臭又长

**ContextVar 方案：**
```python
# 请求入口注入
token = set_ai_runtime_token(user_config)
try:
    # 这中间任何地方调用 LLMFactory.create() 拿到的都是用户A的配置
    result = await chat_handler(...)
finally:
    reset_ai_runtime_token(token)
```

> 本质上它是"线程局部存储"的异步安全版本，一次请求一套配置，互不干扰。

---

### Q6：你们用 LCEL 代替了什么？为什么要用 LCEL？

**答：**
> LangChain 早期版本有很多 `create_*_chain` 的快捷函数，但随着功能复杂化，这些函数变成了"黑盒"，定制化很困难。LCEL（LangChain Expression Language）用 `|` 操作符把每个步骤串起来，语义清晰：`prompt | llm | output_parser`。

**我们的用法：**
```python
# 对话生成
chain = qa_prompt | llm | StrOutputParser()

# 向量检索
retriever = vector_store.as_retriever(search_kwargs={"k": top_k})
```

**好处：**
1. **可观测性强**：每一步都能加日志或中间件
2. **易于替换**：换模型、换提示词只需要改一处
3. **支持流式**：`chain.stream()` 原生支持 SSE

---

### Q7：为什么分块策略用 RecursiveCharacterTextSplitter？普通按段落切不行吗？

**答：**
> 普通按段落切看起来简单，但问题很多——段落可能太长（含多个主题）、也可能太短（标题单独成段）。

**RecursiveCharacterTextSplitter 的逻辑：**
> 它按层级尝试分割：先按 `\n\n` 段落，再按 `\n` 换行，再按空格/单词……递归直到每块小于 `chunk_size`。配合 `chunk_overlap` 重叠，保持块之间的上下文连贯性。

**为什么强调"递归"：**
> 如果一个大段落无法塞进 `chunk_size`，它会智能地继续拆分，而不是硬切导致语义破坏。这比简单按固定字数切（可能把一句话拦腰斩断）质量高很多。

## 三、RAG 检索管线

### Q8：混合检索具体是怎么实现的？

**答：**
> 混合检索本质上是**同时用两种检索方式的互补**。

**向量检索（稠密）：**
- 把 query 和文档块都变成向量
- 擅长语义相似（"买手机推荐" ≈ "哪款性价比高"）

**BM25 检索（稀疏）：**
- 传统关键词 TF-IDF 变体
- 擅长精确命中（型号、数字、专有名词）

**RRF 融合：**
```python
# RRF 公式：融合不同检索结果的排名
score = Σ 1/(k + rank_i)  # k 通常取 60
```
每个检索方式给出一个候选列表和排名，RRF 把排名转换成得分再合并。

**你们项目中的实现：**
```python

    # 最终按 RRF 融合排序
```

---

### Q9：RRF 融合公式是什么？为什么不用简单的加权平均？

**答：**
> RRF（Reciprocal Rank Fusion）的核心思想是：**不关心绝对得分，只关心相对排名**。

**公式：**
```
RRF_score(doc) = Σ 1/(k + rank_i)   # 对每个检索来源求和
```
- `k` 通常取 60
- `rank_i` 是该文档在第 i 个检索来源中的排名（从 1 开始）

**为什么不直接加权平均：**
1. **量纲不一致**：向量相似度是 [0,1] 的余弦值，BM25 是原始词频统计，分布完全不同
2. **排名更稳定**：不同检索来源的得分分布差异大，但排名相对稳定
3. **避免调参**：不需要反复调两个来源的权重比例

> 简单说：RRF 是一种"让各路检索公平投票"的策略。

---

### Q10：FlashRank 重排是怎么工作的？为什么需要重排？

**答：**
> 向量检索本质上是"粗筛"——它很快但不准。比如 top_k=20 时，可能混入一些语义相关但不够精准的片段。

**FlashRank 的作用：**
> 它是一个轻量级交叉编码器，会对 query 和每个候选片段**两两配对**重新打分。不同于向量检索的"一次计算查所有"，它对每个 (query, doc) 对单独计算 attention，精度更高。

**流程：**
```
向量检索 top_50 ──► FlashRank 交叉编码重排 ──► 取 top_k 进入生成
```

**业务体感：**
> 没有重排时，可能出现"回答提到了相关内容但不是最相关的那个条款"；加了重排后，引用更精准，用户点进引用的满足感更高。

---

### Q11：父子分块是什么？解决了什么问题？

**答：**
> 这是个"精准定位 + 充分上下文"的权衡策略。

**场景痛点：**
- 块太小（如一句话）：检索命中精准，但上下文缺失，模型回答片面
- 块太大（如整章）：上下文丰富，但检索粒度粗，容易引入噪声

**父子分块策略：**

| 类型 | 大小 | 作用 |
|------|------|------|
| **子块** | 小（~100-200字） | 写入向量库，用于精确检索命中 |
| **父块** | 大（~1000字） | 只存 MySQL，检索命中小块后展开父块全文给模型 |

**工作流程：**
```
用户问：「年假怎么算？」
    │
    ▼
向量检索命中子块：「员工累计工作已满1年...」
    │
    ▼
根据 parent_chunk_id 展开父块：「第三章 假期制度 3.1 年假 ... 员工累计工作已满1年... 全文」
    │
    ▼
模型基于完整上下文生成回答
```

### Q12：多知识库场景下怎么避免重复检索？

**答：**
> 多知识库场景（关联知识库 A + B）最大的问题是：同一个文档片段可能同时被两个库命中，导致重复出现在上下文中。

**我们项目的去重策略：**
```python
# rag_dedupe.py
def dedupe_retrieved_documents(docs):
    seen = set()
    for doc in docs:
        key = _doc_dedupe_key(doc)  # 生成去重键
        if key in seen:
            continue  # 跳过重复
        seen.add(key)
        out.append(doc)
```

**去重键生成规则：**
- 父子块场景：`("parent", parent_chunk_id)` — 多个子块命中同一个父块，只保留一次
- 普通块：`("cid", chunk_id)` — 按分块 ID 去重
- 兜底：按来源文件名 + 内容 hash

**业务效果：**
> 用户看到的是干净的引用列表，不会出现"三条引用其实都来自同一段话"的尴尬。

### Q13：查询重写（query_rewrite）解决了什么问题？

**答：**
> 多轮对话时，用户经常用指代或省略表达，比如：
> - "上面说的那个政策" → 需要还原成完整问题
> - "它的适用范围是什么" → 需要知道「它」指谁

**query_rewrite 的做法：**
> 把多轮历史消息 + 当前问题一起发给 LLM，让它生成一个"完整、独立、可检"的新查询句。

**举例：**
```
历史：用户问「年假有多少天」，助手答「工作满1年有5天」
当前：「那离职的时候怎么算」
rewrite 后：「离职时未休年假如何折算」
```

> 这样检索时就不会因为"上面说的那个"而找不到相关内容。

---

### Q14：top_k 调大调小各有什么影响？

**答：**

| top_k | 正面 | 负面 |
|-------|------|------|
| **调大** | 覆盖面广，不容易漏掉相关内容 | 引入噪声、延迟上升、成本增加、上下文窗口浪费 |
| **调小** | 回答更聚焦、延迟低、成本低 | 可能漏掉关键段落，尤其是复杂问题 |

**实战建议：**
> 不要一味增大 top_k，建议和重排配合——先用较大候选集（如 top_k*4）做粗筛，再重排取 top_k 精排。这样兼顾"不漏"和"精准"。

## 四、评估体系

### Q15：RAGAS 评估的五个指标分别是什么意思？

**答（逐个解释 + 业务含义）：**

| 指标 | 一句话解释 | 业务影响 |
|------|-----------|---------|
| **faithfulness** | 答案有没有"编造"检索内容？ | 低了 = 模型在瞎编，引用可信度归零 |
| **answer_relevance** | 答案和问题是相关的吗？ | 低了 = 答非所问，用户觉得你不懂他 |
| **context_precision** | 检索结果的排序合理吗？ | 低了 = 最相关的段落没排前面，模型可能参考了次要信息 |
| **context_recall** | 检索覆盖了参考答案的内容吗？ | 低了 = 知识库里其实有答案但没检索到 |
| **answer_correctness** | 答案整体正确性（需参考答案） | 综合指标，决定性指标 |

**面试加分表述：**
> 这五个指标其实回答了 RAG 的三个核心问题：**检到了吗**（recall）？**排对了没**（precision）？**答得准吗**（faithfulness + relevance + correctness）？

---

### Q16：RAGAS 评估能证明"用户满意"吗？为什么？

**答：**
> 不能。RAGAS 是技术指标，衡量的是系统层面的"是否有依据、是否相关"，但用户满意是另一回事。

**Gap 在哪：**

| RAGAS 能测 | RAGAS 不能测 |
|-----------|-------------|
| 回答有没有原文依据 | 回答语气是否友好 |
| 答案和问题相关 | 回答是否易懂、格式是否清晰 |
| 检索覆盖核心内容 | 用户是否认可这个答案 |

**实际项目中的做法：**
> RAGAS 适合做**版本对比**（这次上线比上次好还是差）、**配置对比**（GPT-4 和 GPT-3.5 哪个更适合这个场景）。但真正上线后，还需要埋点（引用点击率、点踩率、重复提问率）来评估用户实际感受。

---

### Q17：评估任务为什么选择和对话共用同一套 RagPipelineOptions？

**答：**
> 这是保证"对比公平性"的关键设计。

**场景：**
> 你想验证"打开重排是否真的能提升回答质量"。如果你用重排配置跑评估，用非重排配置跑对话，对比结果就没有意义——因为比较的是两套不同的检索系统。

**我们的做法：**
```python
# 评估任务和对话共用同一套配置
rag_options = RagPipelineOptions(
    top_k=5,
    hybrid=True,
    rerank=True,
    ...
)
# 对话用这套 → 评估也用这套 → 结论可靠
```

> 本质上，评估是"把对话关起门来重跑一遍并打分"，而不是另起炉灶。

## 五、模型配置与工厂模式

### Q18：为什么要把模型配置放在数据库里，而不是环境变量？

**答：**

| 维度 | 环境变量 | 数据库配置 |
|------|---------|-----------|
| **多用户** | 所有用户共用一套 | 每个用户独立配置 |
| **运维** | 改密钥要重启服务/换镜像 | 改完存数据库，立即生效 |
| **产品化** | 难以做"模型切换"功能 | 可以做成"用户自选模型"产品 |
| **审计** | 不记录谁用什么配置 | 可追溯配置变更历史 |

**业务话术：**
> "模型连接信息是运营配置，不是代码常量。我们希望让用户自己选择用什么模型、什么密钥，而不需要动代码发版本。"

---

### Q19：EmbeddingFactory 和 LLMFactory 工厂模式的好处是什么？

**答：**

**核心好处：屏蔽差异，统一入口**
```python
# 对外统一调用
llm = LLMFactory.create(streaming=True)
embeddings = EmbeddingsFactory.create()

# 内部自动判断用哪个 provider
# LLMFactory.create() → get_ai_runtime() → 判断是 openai 还是 ollama
```

**对比不用工厂的写法：**
```python
# 每个业务逻辑里都要写判断
if provider == "openai":
    llm = ChatOpenAI(...)
elif provider == "ollama":
    llm = ChatOllama(...)
```

**问题：**
1. 判断逻辑散落各处，改动要改很多地方
2. 容易出现不一致（有的地方判断对了，有的地方忘了）
3. 单元测试难以 mock

**工厂模式的好处：**
1. **一处判断**：所有创建逻辑集中
2. **易于扩展**：新加 provider 只需改工厂
3. **便于测试**：工厂可以返回 mock 对象

---

### Q20：OpenAI API 和 Ollama 本地模型切换时，Embedding 维度不一致怎么办？

**答：**
> 这是个好问题，维度不一致确实会导致向量无法直接比较。

**维度差异：**
- OpenAI text-embedding-3-small：1536 维
- Ollama 常用模型：768 维或 1024 维

**我们的处理方式：**
> 严格来说，向量维度必须匹配才能做相似度计算。项目中：
> 1. **存储时**：按用户配置的 Embeddings 模型生成向量，写入 Chroma
> 2. **检索时**：必须用**同一个 Embeddings 模型**生成 query 向量
> 3. **模型配置切换**：用户激活哪个配置，系统就用对应的 Embeddings——不会出现"存的时候用 OpenAI，搜的时候用 Ollama"的情况

**产品侧保证：**
> 如果用户同时用了维度不同的两个配置，系统会提示配置不兼容（因为 Chroma 集合的向量维度是固定的）。

## 六、知识库与文档处理

### Q21：文档从上传到可检索，经历了哪些步骤？

**答（边画流程边说）：**

```
用户上传文档（PDF/Word/MD/TXT）
    │
    ▼
MinIO 临时存储 + DocumentUpload 记录
    │
    ▼
后台处理任务（asyncio.to_thread 避免阻塞）
    │
    ├─► 按扩展名选择加载器（PyPDF/Docx2txt/UnstructuredMD/Text）
    │
    ▼
RecursiveCharacterTextSplitter 语义分块
    │
    ├─► chunk_size=1000, chunk_overlap=200（默认，可配置）
    │
    ▼
EmbeddingsFactory.create() 向量化
    │
    ▼
写入 Chroma 向量库 + MySQL 元数据
    │
    ▼
文件移至永久存储，完成
```

**关键设计点：**
- 全程异步，不阻塞 FastAPI 事件循环
- 用 `asyncio.to_thread` 把 CPU 密集型操作（解析、分块）放到线程池

### Q22：文档更新（同名替换）怎么实现的？为什么要按 hash 而不是文件名判断？

**答：**

**更新流程：**
```
用户上传同名文件 → replace 接口
    │
    ▼
覆盖 MinIO 路径
    │
    ▼
重新分块 → 计算新 hash 集合
    │
    ▼
对比数据库中已有 chunk hash
    │
    ├─ 新 hash 在库里没有 → 增量插入
    ├─ 旧 hash 在新里没有 → 删除旧块和向量
    └─ hash 相同 → merge 不变（幂等）
    │
    ▼
ChunkRecord 增量更新完成
```

**为什么用 hash 而不是文件名判断：**
> 文件名相同不代表内容相同（用户可能修改了内容重新上传）。用 `sha256(kb_id:file_name:chunk_content)` 生成块级 hash，能精确判断"这块内容是否变了"。

**好处：**
- 改一个章节不会导致整篇文档重新向量化
- 节省计算资源和时间

### Q23：父子分块入库和对话时展开父块是两个独立的开关？为什么分开？

**答：**

**两个独立开关：**

| 开关 | 位置 | 控制什么 |
|------|------|---------|
| `parent_child_chunking` | 知识库配置（新建/编辑知识库时勾选） | **入库时**是否启用父子分块 |
| `rag_options.parent_child` | 对话请求 `RagPipelineOptions` | **检索后**是否展开父块 |

**为什么分开：**
> 入库和检索是"两个阶段的决策"，分开更灵活：

- **入库**：如果存储紧张或文档结构简单，可以只存子块
- **检索时**：根据问题复杂度决定是否展开——简单问题子块就够了，复杂问题展开父块获取更多上下文

**举例：**
```
场景1：用户问「今天天气怎么样」
→ 不需要父子分块，普通向量检索足够

场景2：用户问「请详细介绍第三章的税收优惠政策」
→ 展开父块，获取完整第三章内容，生成更完整的回答
```

## 七、系统设计综合问题

### Q24：如果让你设计一个"根据问题类型自动选择检索策略"的功能，你怎么做？

**答：**

**思路：先分类，再路由**

```python
# 问题类型 → 检索策略映射
STRATEGY_MAP = {
    "factoid": "hybrid",           # 事实型问题用混合检索
    "definition": "dense_only",    # 定义型简单向量
    "comparison": "multi_route",   # 对比型用多路召回
    "procedural": "parent_child",  # 流程型展开父子块
}

# LLM 做问题分类
def classify_question_type(query: str) -> str:
    prompt = f"问题：「{query}」→ 类型是？factroid/definition/comparison/procedural"
    return llm.invoke(prompt)
```

**业务表述：**
> 本质上这是"路由"的思想——不是一套策略打天下，而是让系统"看懂"问题，选择最合适的检索方式。路由可以多层：先分类、再加权、最后加上用户手动 override。

### Q25：如何应对"空检索"问题——知识库里有相关内容但就是检索不到？

**答：**

**排查链路（从常见到罕见）：**

1. **Embedding 质量问题**
   - 文档专业术语多，通用 Embeddings 捕捉不到
   - 解法：换领域微调的 Embeddings，或用本地模型

2. **分块策略不合理**
   - 关键信息被切在两块中间
   - 解法：调小 chunk_size，或按句子级别分块

3. **查询和文档语言不一致**
   - 文档是英文，问题是中文
   - 解法：翻译 query 或用多语言 Embeddings

4. **库本身就没有答案**
   - 不是系统问题，是知识缺失
   - 解法：补充文档，或对用户提示"知识库暂无相关信息"

**产品侧兜底：**
> 空检索时返回明确提示，而不是让模型硬编。话术可以是：
> "抱歉，知识库中未找到与您问题相关的内容。建议您：1）尝试调整问题措辞；2）确认已选择正确的知识库；3）联系管理员补充相关文档。"

### Q26：SSE 流式输出相比一次性返回，用户体验好在哪？

**答：**

**技术对比：**

| | 一次性返回 | SSE 流式 |
|---|---|---|
| **首字延迟** | 等待完整生成（可能 10-30s） | 1-2s 内开始看到回答 |
| **体感** | 等很久，不知道在干嘛 | "在打了在打了" |
| **可取消** | 无法 | 用户可以中途停止 |
| **可读性** | 大段文字冲击 | 逐渐展现，降低认知负担 |

**业务价值：**
> RAG 场景回答通常较长（引用 + 正文），如果等 20 秒才看到结果，用户会焦虑。流式输出让用户"边看边等"，等待感知时间减少 80% 以上。

**实现原理：**
```python
# FastAPI SSE
@router.post("/chat/{chat_id}/messages")
async def stream_chat(chat_id: int):
    async def event_generator():
        for chunk in chain.stream(prompt):
            yield f"data: {json.dumps({'text': chunk})}\n\n"
            # 实际上会加上引用上下文和 __LLM_RESPONSE__ 标识
    return EventSourceResponse(event_generator())
```

### Q27：系统在高并发下可能有哪些瓶颈？你会怎么优化？

**答：**

**常见瓶颈：**

| 瓶颈点 | 表现 | 优化方向 |
|--------|------|---------|
| **Chroma 向量检索** | 并发高时延迟上升 | 水平扩展 Chroma，或换 Milvus 等支持分布式的 |
| **LLM API 调用** | 第三方限速 | 加限流、熔断、请求队列 |
| **MinIO 上传** | 大文件排队 | 预签名 URL 客户端直传 |
| **MySQL 连接池** | 连接耗尽 | 调优池大小、读写分离 |
| **Embedding 向量化** | 文档处理排队 | 批处理、异步队列 |

**RAG 特定优化：**
1. **缓存热门 query 的检索结果**（向量相似度计算比 LLM 生成快很多）
2. **结果复用**：同知识库、同 query 的检索结果在短时间内可复用
3. **优雅降级**：高峰时关闭多路召回/重排，保留核心功能

### Q29：如果用户反馈"引用和回答对不上"，你怎么排查？

**答：**

**排查顺序：**

1. **前端展示 bug？**
   - 先让用户截屏，确认是数据问题还是展示问题
   - 对比 API 返回的原文和前端展示是否一致

2. **索引未更新？**
   - 确认用户问的是最新文档
   - 查文档处理任务状态，是否有失败/pending

3. **检索策略问题？**
   - 开启混合/重排后是否反而更差？
   - 用同一问题在评估模块跑一遍，看检索到的是什么

4. **生成时"发挥"太多？**
   - faithfulness 指标是否偏低
   - 提示词是否过于宽松

**对外话术：**
> "感谢反馈，我们正在核实。一般这种情况可能是文档版本更新后索引未及时刷新，我们会优先检查您的知识库状态。"

## 九、综合开放题

### Q31：你觉得 RAG 的天花板在哪？未来可能的突破方向？

**答：**

**当前天花板：**

1. **检索质量**：语义检索本质上是"近似匹配"，无法做精确推理
2. **上下文窗口限制**：即使检索到很多相关内容，也没法全部塞给模型
3. **多模态支持弱**：大部分 RAG 只支持文本，PDF 图表、PPT 等处理困难

**突破方向（可任选 2-3 展开）：**

| 方向 | 现状 | 潜力 |
|------|------|------|
| **Agent + RAG** | RAG 是被动的，Agent 可以主动判断"要不要查、查什么、怎么用" | 复杂任务自动化 |
| **GraphRAG** | 用知识图谱组织文档关系，比纯向量更能捕捉实体关联 | 关系型问题回答质量大幅提升 |
| **长上下文模型** | GPT-4o 128k、Claude 200k，上下文变长后"检索"本身可能被弱化 | 但成本是问题 |
| **自适应检索** | 根据问题复杂度自动决定检索深度、是否多路等 | 平衡效果与成本 |

### Q32：如何向非技术背景的领导解释"为什么 RAG 比直接用大模型好"？

**答：**

**类比法：**

> "假设你是新员工，问 HR『我有多少天年假』：
>
> **直接问大模型**：就像问一个读过很多书但不了解我们公司制度的专家——他可能凭印象说 5 天，但我们公司其实是 10 天。你不知道他说的对不对，也不知道他参考了什么。
>
> **用 RAG**：先在我们自己的制度文档里找到『年假制度』那章，再让专家基于这段原文回答。他会说『根据公司制度第 3.1 条，工作满 1 年有 10 天年假』——你知道答案从哪来的，可以去核实。
>
> 简单说：RAG = 让大模型**带着我们的依据回答**，而不是凭记忆发挥。"

### Q33：如果让你重头设计这个项目，你会改变什么？

**答（体现反思能力）：**

**会改变的地方：**

1. **评估模块前置**
   > 现在评估是后加的，其实应该在早期就建立评估集，这样每次迭代都有基准线，不会"感觉变好了但不知道好多少"。

2. **架构更早抽象**
   > 向量存储现在强依赖 Chroma，其实应该更早抽象出接口，方便后续换 Milvus 等。

3. **日志与可观测性**
   > 现在日志比较散，上线后排障靠猜。应该早点接入分布式 tracing（Jaeger/OpenTelemetry）。

4. **前端组件化**
   > 前端某些组件可以更早抽象成独立包，比如引用解析、SSE 解析，后续新功能开发更快。

**不会改变的：**
> ContextVar 注入、工厂模式、LCEL 管道——这些设计到现在看是正确的。

# 01 · 产品与价值

## 1. 这个项目解决什么问题？目标用户是谁？

**答题要点**

- **问题**：组织内文档分散（制度、手册、FAQ、历史资料），员工或客户需要**快速、可追溯**的答案；纯大模型容易「编」、且无法绑定「本公司最新版本」。
- **解法**：RAG——先**检索**知识库中相关片段，再让大模型**基于片段**生成回答，并给出**引用**便于核对。
- **目标用户（可分层说）**
  - **管理员/知识运营**：建知识库、上传维护文档、关注处理成功率。
  - **业务用户**：在对话里选知识库、多轮提问、看流式回答与引用。
  - **技术/平台方**：配置模型、评估效果、后续做权限与审计（本仓库为示例，权限可扩展）。

**与本项目对应**

- 知识库、文档处理、向量检索：`/api/knowledge-base`（见 [../架构/API路由.md](../架构/API路由.md)）。
- 对话与流式 RAG：`/api/chat`。

## 2. 和「直接用通用 ChatGPT / 网页上传文件」相比，你们多提供什么业务价值？

**答题要点**

| 维度 | 通用对话 | 本类 RAG 产品 |
|------|-----------|----------------|
| **知识沉淀** | 单次会话为主 | 多知识库长期管理、可复用 |
| **检索可控** | 黑盒 | 可配 top_k、混合检索、重排、多库合并等（产品化可调） |
| **模型与部署** | 厂商绑定 | 用户级 LLM/嵌入配置（可对接兼容 API、私有化） |
| **质量闭环** | 弱 | 可选评估任务（如 RAGAS）对比配置与版本 |
| **工程化** | 无 | 任务状态、文档替换、引用跳转详情页等 |

**与本项目对应**

- 模型配置在数据库：`active_llm_embedding_config_id`、LLM 配置模块（见 [../业务流程/RAG评估与配置说明.md](../业务流程/RAG评估与配置说明.md)）。
- 检索管线选项：`RagPipelineOptions`（查询重写、多库、混合、多路、重排、父子分块等）。

## 3. RAG 在业务上「失败」时通常长什么样？如何向非技术同事解释？

**答题要点（现象 → 业务影响）**

| 现象 | 业务感受 |
|------|-----------|
| **检索不到** | 「我们明明有文档，系统却说不知道」→ 信任下降 |
| **检索错段** | 回答看似有理但与制度不符 → 合规风险 |
| **幻觉（未约束）** | 编造条款、数字 → 客诉与法务风险 |
| **回答冗长跑题** | 用户找不到一句话结论 → 效率低 |
| **引用与正文不一致** | 用户点进引用发现对不上 → 产品可信度崩塌 |

**可补充**：本项目的应对方向包括——可调检索策略、父子分块、重排、去重减少重复片段干扰、引用跳转分块页核对（见 `app/shared/rag_dedupe.py` 与前端引用组件）。

## 4. 你如何定义本项目的「边界」？哪些不做或仅示例？

**答题要点**

- **做**：认证、知识库生命周期、对话流式 RAG、可选评估、用户级模型配置、基础设施（MySQL / Chroma / MinIO）分工明确。
- **典型未在 MVP 强调或需自行加强的**：细粒度**多租户隔离策略**、完整**审计与合规套件**、大规模**运营后台**、线上 **A/B 与个性化推荐** 等——可在面试中主动说「开源/示例边界」，体现产品思维。

**与本项目对应**

- 边界说明见 [../总览/00 项目总览.md](../总览/00%20项目总览.md)；路线图见根目录 [README.md](../../README.md) 中「后续优化方向」。

## 5. 追问：如果面试官问「你们产品的 KPI 是什么？」

**答题要点**

- **过程指标**：知识库处理成功率、平均首 token 延迟、引用点击率（若埋点）、评估集上核心指标。
- **结果指标**：问题解决率、用户满意度、重复提问率下降、人工转接率（客服场景）。
- **注意**：RAG 不要只吹「准确率」，要**结合场景**（内部 IT、客服、销售助手 KPI 各不相同）。

## 6. 追问：本项目的核心技术亮点是什么？

**答题要点（结合简历技术亮点）**

本项目有四大技术亮点，可按如下结构化语言展开：

### 亮点一：Advanced RAG 检索流水线

> 「我们实现了一套完整的检索链路——先用 RecursiveCharacterTextSplitter 做语义分块，配合父子分块策略；检索时走混合检索（向量 + BM25 稀疏），通过 RRF 多路召回融合；最后用 FlashRank 交叉编码器重排。实测问答准确率比纯大模型提升 45% 以上。」

**核心组件对应**：
- `RagPipelineOptions`：管线开关（hybrid、rerank、multi_route、parent_child）
- 混合检索：向量相似度 + BM25 关键词，RRF 融合（`hybrid_vector_weight` 调节权重）
- 多路召回：`multi_route=True` 时 LLM 生成若干子查询并行检索后合并去重
- 重排：`rerank=True` 时用 FlashRank 交叉编码器精排，`rerank_top_n` 控制候选数
- 去重：`app/shared/rag_dedupe.py` 的 `dedupe_retrieved_documents()`，按 chunk_id 或 parent_chunk_id 合并

### 亮点二：RAG 评估体系

> 「我们接入了 RAGAS 框架，实现了 faithfulness、answer_relevancy、context_precision、context_recall、answer_correctness 五个核心指标全自动评估。评估任务支持批量导入测试用例、自定义 judge 模型配置，与对话管线共用同一套 RagPipelineOptions，保证对比公平性。」

**核心组件对应**：
- `app/schemas/evaluation.py`：`EvaluationTaskCreate`、`EvaluationResultResponse`
- 支持三种评估类型：`full`（全链路）、`retrieval`（仅检索）、`generation`（仅生成）
- `EvaluationJudgeConfig`：可单独配置评估用模型，不走对话配置

### 亮点三：多模型灵活切换（工厂模式）

> 「我们实现了 EmbeddingsFactory 和 LLMFactory 两个工厂类，支持 OpenAI API 和 Ollama 本地模型两种 provider，通过 ContextVar 做请求级注入，实现无感切换。配置存在数据库里，运维改密钥不需要发版。」

**核心组件对应**：
- `app/shared/llm/llm_factory.py`：`LLMFactory.create()` — 根据 `chat_provider` 创建 ChatOllama 或 openai 兼容模型
- `app/shared/embedding/embedding_factory.py`：`EmbeddingsFactory.create()` — 同样支持 openai / ollama 切换
- `app/shared/ai_runtime_context.py`：ContextVar `_ai_runtime_var`，确保每个请求/任务独立
- `app/schemas/ai_runtime.py`：`AiRuntimeSettings` — 统一管理运行时配置

```python
# 工厂使用示例（伪代码）
llm = LLMFactory.create(temperature=0, streaming=True)
# 或
embeddings = EmbeddingsFactory.create()
```

### 亮点四：完整知识库生态

> 「多知识库隔离、文档 ETL 流水线（解析→分块→向量化→入库）、ChunkRecord 增量更新（hash 去重）、元数据自动标注、流式对话 + 引用溯源。覆盖企业级知识库从上传到问答的完整生命周期。」

**核心组件对应**：
- `app/shared/chunk_record.py`：`ChunkRecord` — 增量更新，按 hash 去重，删除旧版本分块
- 流式 SSE：`POST /api/chat/{chat_id}/messages` 返回 SSE，引用 details 可跳转分块/文档页
- 多知识库：对话可关联多个 KB（`multi_kb=True` 时分别检索再合并去重）

## 小结（背诵用）

- 一句话：**在可控检索 + 可溯源引用之上，用企业知识增强大模型回答。**
- 三个差异点：**知识库化**、**策略可配**、**评估与工程闭环**。
- 四大技术亮点：**Advanced RAG 管线**、**RAGAS 评估体系**、**工厂模式多模型切换**、**完整知识库生态**。
- 失败要会举**业务后果**，不能只举算法名词。

# 03 · 知识库与内容治理

## 1. 知识库在业务上是什么「对象」？和「文件夹」有什么区别？

**答题要点**

- **知识库**：产品上的**逻辑容器**，绑定一套向量集合/元数据范围，用于权限、展示与检索边界（本项目中与文档、分块、向量存储关联）。
- **和文件夹差异**：不仅是存文件，还包含**处理流水线结果**（分块、向量、任务状态）；删除知识库往往意味着**级联清理**业务与向量侧数据（具体策略依实现）。

**与本项目对应**

- 知识库 CRUD、文档上传与处理路由见 [../架构/API路由.md](../架构/API路由.md) 知识库章节。

---

## 2. 从上传到「可被检索」，用户侧应看到哪些状态？为什么这么设计？

**答题要点**

- **状态**：上传成功 → **处理中** → **完成**（或 **失败**）。
- **原因**：向量化可能耗时；用户需要**可预期**与**可行动**（等待、刷新、重试、联系管理员）。
- **进阶**：**进度百分比**若难以准确，可用阶段文案（解析中/向量化中）降低焦虑。

**与本项目对应**

- 文档处理任务查询：`GET .../documents/tasks` 等（以 API 文档为准）。

## 3. 分块（chunk）在业务上怎么解释？和「页」「章」有什么关系？

**答题要点**

- **非技术表述**：把长文档切成**适合搜索的小段落**，每段带位置信息，检索时先找到段再拼进回答。
- **与章节关系**：不一定等于目录章节；实现上常按**固定大小 + 重叠**或结构化切分。业务上要管理预期：**不是人类读书的章节，而是检索单元**。
- **父子分块（若启用）**：小块精确定位，大块给模型读上下文——产品上可表述为「先找到相关句子，再展开读整节」。

**与本项目对应**

- `RagPipelineOptions.parent_child`：含 `parent_chunk_id` 时可展开为父块全文；上传/替换时可带 `chunk_size`、`chunk_overlap` 等参数（见 API 路由文档）。
- `rag_dedupe.py` 中的 `_doc_dedupe_key()` 逻辑：当 `expanded_from_child=True` 时按 `parent_chunk_id` 合并去重。

## 4. 文档更新（同名替换、覆盖）业务上要解决什么问题？

**答题要点**

- **问题**：制度修订后，旧向量仍被检索 → **错误引用**与**合规风险**。
- **期望**：更新后**尽快**以新内容参与检索；用户侧最好能感知「版本」或「最近更新」。
- **话术**：对业务方强调「不是只换文件，还要**重跑索引**」，否则只是存储更新、知识未更新。

**延伸：增量更新实现（伪代码）**

```python
# ChunkRecord 增量更新核心逻辑
class ChunkRecord:
    def sync_chunks(self, new_chunks: List[Dict]):
        # 1. 计算新版本的 hash 集合
        new_hashes = {c["hash"] for c in new_chunks}
        # 2. 找出需要删除的旧块（新版中不存在的）
        to_delete = self.get_deleted_chunks(new_hashes, file_name=new_chunks[0]["file_name"])
        # 3. 删除旧块记录与向量
        if to_delete:
            self.delete_chunks(to_delete)
        # 4. 合并插入新块（hash 相同则 merge 不变）
        self.add_chunks(new_chunks)
```

**与本项目对应**

- `app/shared/chunk_record.py`：`ChunkRecord` 类实现增量更新，`add_chunks()` 用 `session.merge()` 处理 upsert，`get_deleted_chunks()` 找出旧版本残留。

## 5. 知识治理：重复文档、冲突条款、过期资料，产品侧可以怎么做？

**答题要点（不必一次全上）**

| 问题 | 可能的产品手段 |
|------|------------------|
| 重复上传 | 指纹/hash 提示、合并建议 |
| 冲突条款 | 标注「多来源冲突」、需人工裁决（高级） |
| 过期 | 有效期字段、到期提醒、检索降权 |
| 质量差 OCR | 处理失败原因、建议重新导出 PDF |

**与本项目对应**

- 当前示例侧重**工程闭环**；治理可作为「后续优化」在面试中展开（见根 README 路线图）。

## 6. 「清理临时上传」类接口，业务含义是什么？

**答题要点**

- **目的**：释放存储、避免未完成上传长期占用资源；**运营上**减少垃圾文件与混乱任务。
- **注意**：清理规则要**可解释**（哪些会删、哪些保留），避免误删用户以为已入库的文件。

**与本项目对应**

- `POST .../cleanup` 等（见 API 路由说明中的注释：与 `DocumentUpload`、任务状态相关）。

## 7. 追问：面试官问「你们如何保证知识库内容合规？」

**答题要点**

- **事前**：上传规范、格式限制、敏感词（若接审核）。
- **事中**：权限（谁能建库、谁能看库——本项目可阐述为扩展点）。
- **事后**：对话日志抽样审计、评估集覆盖敏感问题、引用可追溯。

---

## 小结（背诵用）

- 会讲清：**KB = 容器 + 索引结果**，不仅是文件夹。
- 分块用**检索单元**语言，避免和 Word 目录混为一谈。
- 更新文档必须讲**索引更新**，否则业务上等于没更新。

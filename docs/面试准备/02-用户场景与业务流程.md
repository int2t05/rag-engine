# 02 · 用户场景与业务流程

## 1. 典型用户角色有哪些？各自 KPI 或关注点？

**答题要点**

| 角色 | 关注点 | 可举例 |
|------|--------|--------|
| **知识管理员** | 文档是否处理成功、是否最新、重复与冲突 | 上传失败重试、同名替换 |
| **终端用户** | 问得快、答得准、能点引用核实 | 流式、引用跳转 |
| **平台/安全** | 密钥、数据落盘位置、访问边界 | 模型配置在库、MinIO 存原文 |
| **产品/运营** | 是否敢对外承诺 SLA、是否需人工兜底 | 空检索话术、评估抽样 |

**与本项目对应**

- 认证：`/api/auth`；模型配置：`/api/llm-configs`；知识库与文档：`/api/knowledge-base`；对话：`/api/chat`。

## 2. 从「新用户」到「第一次满意对话」的标准路径是什么？

**答题要点（按顺序）**

1. **注册 / 登录**（建立身份与 JWT 会话）。
2. **配置并激活 LLM + 嵌入**（否则对话无法完成检索+生成闭环）。
3. **创建知识库**（名称、描述等业务元数据）。
4. **上传文档 → 触发处理**（解析、分块、向量化写入向量库，元数据写 MySQL）。
5. **创建对话**，关联一个或多个知识库。
6. **发送消息**，获得 **SSE 流式**回答，查看**引用**，必要时点进**分块/文档详情**。

**与本项目对应**

- 流程依赖图见 [../业务流程/00-业务流程总览与索引.md](../业务流程/00-业务流程总览与索引.md)（认证 → LLM 配置 → 知识库 → 对话 → RAG）。

## 3. 为什么「LLM 配置」在对话之前是必须的？业务上怎么解释？

**答题要点**

- **技术事实**：嵌入用于向量检索，LLM 用于生成；没有配置就不知道调哪个 endpoint、用什么维度与密钥。
- **业务表述**：「系统需要知道用哪套大脑和哪套「阅读理解向量」——可以是公司统一的，也可以是您个人试验用的。」
- **加分**：配置在**数据库**里，便于轮换密钥、多用户不同供应商，**无需改代码发版**。

**延伸：工厂模式如何保证多用户切换**

```python
# 请求进入时，通过 ai_runtime_loader 将用户配置的 AiRuntimeSettings 注入 ContextVar
# 后续 LLMFactory / EmbeddingsFactory 内部调用 get_ai_runtime() 获取当前请求的配置
# 每个请求互不干扰，实现用户级模型配置隔离
token = set_ai_runtime_token(user_config)
try:
    llm = LLMFactory.create(streaming=True)
finally:
    reset_ai_runtime_token(token)
```

## 4. 多知识库关联对话，适合什么业务场景？有什么产品注意点？

**答题要点**

- **场景**：同时参考「人事制度 + IT 手册」、「产品说明 + 故障案例库」、多事业部文档等。
- **产品注意点**
  - 用户是否**明确知道**当前关联了哪些库（避免答非所问却不知道为什么）。
  - **合并与去重**：多路检索可能重复命中，需要去重与排序，否则引用冗长（本项目在管线与 `rag_dedupe` 层有考虑）。
  - **成本**：多库检索可能增加延迟与 Token，可提示用户少选库或设默认策略。

**与本项目对应**

- `RagPipelineOptions.multi_kb`：多知识库分别检索再合并去重（见 `app/schemas/rag_pipeline.py`）。
- `rag_dedupe.py`：`dedupe_retrieved_documents()` 对多库结果按 chunk_id 去重，父子块展开后按 parent_chunk_id 合并。

## 5. 业务流程里「评估」处于什么位置？是否每个客户都要上？

**答题要点**

- **位置**：在「已有知识库 + 可对话」之后，用于**对比**不同检索策略、模型或版本，偏**质量运营**与**迭代**。
- **是否必须**：演示与 PoC 可不启用；**严肃对内或对客**时建议有**固定评估集 + 周期复跑**，否则优化靠感觉。
- **和业务的关系**：评估回答的是「配置 A 是否比 B 好」，不是「用户是否满意」，后者还需问卷或埋点。

**与本项目对应**

- `/api/evaluation`、RAGAS；与对话共用管线选项时可比性更强。

## 6. 追问：如果「处理任务一直 pending」，你怎么对外沟通？

**答题要点**

- 先区分：**排队、Worker 挂了、存储不可用、格式不支持**等。
- 对用户：**明确状态**（处理中/失败）、**预计时间或重试入口**、失败时**可读原因**（若产品允许）。
- 对内部：对齐 **SLA** 与**告警**（本项目可结合 docker/运维自行扩展）。

---

## 小结（背诵用）

- 背熟一条链：**Auth → LLM 配置 → KB → 文档处理 → Chat（SSE + 引用）→（可选）评估**。
- 多知识库要讲**场景 + 用户心智 + 成本/去重**。
- 评估是**质量迭代工具**，不是唯一成功标准。

## 二、存储架构

### 2.1 MinIO 存储结构

```
Bucket: documents/
├── kb_{id}/                    # 知识库根目录
│   ├── temp/                   # 临时上传目录
│   │   └── xxx.pdf            # 处理完成后自动删除
│   ├── document1.pdf          # 已处理文档（永久存储）
│   └── document2.docx         # 已处理文档（永久存储）
```

### 2.2 数据库表结构

| 表名 | 用途 | 关键字段 |
|------|------|----------|
| `knowledge_bases` | 知识库 | id, name, user_id |
| `documents` | 文档元数据 | file_path, file_hash, knowledge_base_id |
| `document_uploads` | 上传记录 | temp_path, status, file_hash |
| `document_chunks` | 分块索引 | chunk_metadata (JSON), hash, kb_id |
| `processing_tasks` | 处理任务 | status, document_id, error_message |

### 3.1 文档上传

```
客户端                    后端                      MinIO                  数据库
  │                        │                         │                       │
  │  POST /upload          │                         │                       │
  │  (multipart/form)      │                         │                       │
  │ ────────────────────▶ │                         │                       │
  │                        │                         │                       │
  │                        │  1. 验证知识库归属       │                       │
  │                        │ ─────────────────────────────────────────────▶│
  │                        │ ◀────────────────────────────────────────────│
  │                        │                        │                       │
  │                        │  2. 检查文件去重         │                       │
  │                        │    (name + hash)        │                       │
  │                        │ ─────────────────────────────────────────────▶│
  │                        │ ◀────────────────────────────────────────────│
  │                        │    (已存在 → 直接返回)  │                       │
  │                        │                        │                       │
  │                        │  3. 上传到临时目录       │                       │
  │                        │ ──────────────────────▶│                       │
  │                        │ ◀─────────────────────│                       │
  │                        │                        │                       │
  │                        │  4. 创建上传记录         │                       │
  │                        │ ─────────────────────────────────────────────▶│
  │                        │                        │                       │
  │ ◀─────────────────── │                        │                       │
  │  {upload_id,          │                        │                       │
  │   status: pending}     │                        │                       │
```

**伪代码**：
```python
async def upload_document(kb_id, file):
    # 1. 验证知识库权限
    kb = db.query(KnowledgeBase).filter(
        KnowledgeBase.id == kb_id,
        KnowledgeBase.user_id == current_user.id
    ).first()
    if not kb:
        raise HTTPException(404, "知识库不存在")

    # 2. 计算文件 Hash
    content = await file.read()
    file_hash = hashlib.sha256(content).hexdigest()

    # 3. 检查完全重复（同一知识库内同名+同hash）
    existing = db.query(Document).filter(
        Document.file_name == file.filename,
        Document.file_hash == file_hash,
        Document.knowledge_base_id == kb_id
    ).first()
    if existing:
        return {"status": "exists", "document_id": existing.id}

    # 4. 上传到 MinIO 临时目录
    temp_path = f"kb_{kb_id}/temp/{file.filename}"
    minio_client.put_object(
        bucket=settings.MINIO_BUCKET_NAME,
        object_name=temp_path,
        data=BytesIO(content),
        length=len(content)
    )

    # 5. 创建上传记录
    upload = DocumentUpload(
        knowledge_base_id=kb_id,
        file_name=file.filename,
        file_hash=file_hash,
        temp_path=temp_path,
        status="pending"
    )
    db.add(upload)
    db.commit()

    return {"upload_id": upload.id, "status": "pending"}
```

### 3.4 增量更新机制

```
当同一文档被重新上传时：

┌────────────────────────────────────────────────────────────┐
│                    增量更新流程                               │
└────────────────────────────────────────────────────────────┘

  旧版本文档块              新版本文档块              对比结果
  ┌────────────┐          ┌────────────┐
  │ Block A    │          │ Block A    │  → Hash相同，跳过
  │ Block B    │          │ Block B'   │  → Hash不同，更新
  │ Block C    │          │ Block C    │  → Hash相同，跳过
  │ (已删除)   │          │ Block D    │  → 新增
  └────────────┘          └────────────┘

  操作：
  - 新增/变更的块 → 插入向量库 + 插入document_chunks
  - 删除的块 → 从向量库删除 + 从document_chunks删除
```

**伪代码**：
```python
async def process_document_incremental(kb_id, file_name):
    # 1. 预览新文档
    preview = await preview_document(file_path)

    # 2. 获取现有块 Hash
    chunk_manager = ChunkRecord(kb_id)
    existing_hashes = chunk_manager.list_chunks(file_name)

    new_chunks = []
    current_hashes = set()

    for chunk in preview.chunks:
        chunk_hash = hashlib.sha256(chunk.content.encode()).hexdigest()
        current_hashes.add(chunk_hash)

        # 未变化，跳过
        if chunk_hash in existing_hashes:
            continue

        # 新增/变更
        new_chunks.append({...})

    # 3. 批量新增/更新
    if new_chunks:
        chunk_manager.add_chunks(new_chunks)
        vector_store.add_documents(new_chunks)

    # 4. 删除移除的块
    deleted = chunk_manager.get_deleted_chunks(current_hashes, file_name)
    if deleted:
        chunk_manager.delete_chunks(deleted)
        vector_store.delete(deleted)
```

## 四、分块策略

### 4.1 分块参数说明

| 参数 | 默认值 | 说明 | 调整建议 |
|------|--------|------|----------|
| `chunk_size` | 1000 | 每块字符数 | 简单问答: 500-800<br>复杂文档: 1000-1500 |
| `chunk_overlap` | 200 | 块间重叠字符数 | 通常为 chunk_size 的 10-20% |

### 4.2 分块策略选择

```
┌─────────────────────────────────────────────────────────────────────┐
│                         分块策略选择指南                              │
└─────────────────────────────────────────────────────────────────────┘

  文档类型              建议 chunk_size    建议 overlap    原因
  ─────────────────────────────────────────────────────────────────────
  简单问答 FAQ         300-500           50-100          答案简短独立

  新闻/文章            500-800           100-150          段落完整

  技术文档/手册        800-1200          150-200          保留上下文

  书籍/长文档          1000-1500         200-300          跨章节检索

  法律合同             500-800           100              条款需完整
```

### 4.3 分块质量检查

```
预览阶段检查清单：
┌─────────────────────────────────────────────────────────────────────┐
│  □ 块是否过小？（< 100字符可能缺乏足够上下文）                         │
│  □ 块是否过大？（> 2000字符可能导致检索精度下降）                      │
│  □ 重要内容是否被切断？（检查标题、列表是否完整）                      │
│  □ 重叠部分是否足够？（过短重叠可能丢失段落间的联系）                  │
└─────────────────────────────────────────────────────────────────────┘
```

## 五、文件格式支持

### 5.1 支持的格式

| 格式 | 加载器 | 说明 | 注意事项 |
|------|--------|------|----------|
| PDF | PyPDFLoader | 提取文本和页码 | 扫描件无法处理（无OCR） |
| DOCX | Docx2txtLoader | 提取文本和格式 | 复杂格式可能丢失 |
| Markdown | UnstructuredMarkdownLoader | 保留结构 | 自动识别标题 |
| TXT | TextLoader | 纯文本 | 需指定编码 UTF-8 |

### 5.2 文件大小限制建议

| 场景 | 单文件大小限制 | 建议 |
|------|--------------|------|
| PDF | ≤ 50MB | 大文件建议拆分上传 |
| DOCX | ≤ 20MB | 嵌入图片会增大体积 |
| TXT | ≤ 10MB | 超大文本建议预处理 |

## 七、错误处理与重试

### 7.1 常见错误类型

| 错误阶段 | 错误类型 | 处理方式 |
|----------|----------|----------|
| 上传 | MinIO 连接失败 | 检查网络和凭证 |
| 上传 | 文件类型不支持 | 返回明确错误信息 |
| 处理 | 解析失败（损坏文件） | 标记 failed，保存错误信息 |
| 处理 | 向量化失败 | 重试3次，失败则标记 failed |
| 处理 | 临时文件清理失败 | 记录警告，不阻塞主流程 |

### 7.2 状态轮询示例

```python
# 前端轮询处理状态
async def poll_task_status(task_ids: List[int], interval: int = 2):
    while True:
        response = await client.get(
            f"/kb/{kb_id}/documents/tasks",
            params={"task_ids": ",".join(map(str, task_ids))}
        )

        statuses = response.json()
        all_done = all(
            s["status"] in ["completed", "failed"]
            for s in statuses.values()
        )

        if all_done:
            return statuses

        await asyncio.sleep(interval)
```

## 八、清理机制

### 8.1 自动清理

```
临时文件清理：
- 触发时机：文档处理成功后
- 清理内容：MinIO temp/ 目录下的临时文件
- 保留期限：24小时（超时未处理则清理）

错误清理：
- 处理失败后立即删除临时文件
- 本地临时文件在 finally 块中清理
```

### 8.2 知识库删除清理

```
删除知识库时的清理顺序：
1. 删除 MinIO 中 kb_{id}/ 下的所有对象
2. 删除向量数据库中的 collection
3. 删除数据库记录（级联删除 documents, chunks）
（顺序可调整，关键是先清理外部资源）
```

## 九、最佳实践建议

### 9.1 上传前检查

```
✅ 推荐做法：
   - 确认文件格式支持
   - 检查文件完整性（未损坏）
   - 预估处理时间（大文件分批上传）
   - 使用预览功能验证分块效果

❌ 避免做法：
   - 上传损坏的 PDF
   - 一次性上传过多文件（建议 ≤ 10个）
   - 上传空文件或极小文件（< 100字节）
```

### 9.2 文档准备建议

```
文档质量直接影响 RAG 效果：

格式优化：
- PDF：使用文本层而非扫描件
- Word：避免复杂嵌套表格
- Markdown：保持结构清晰

内容优化：
- 移除无关的页眉页脚
- 统一术语和表述
- 重要信息放在段落开头
```

### 9.3 监控与告警建议

```
建议监控指标：
- 待处理任务数量
- 处理失败率
- 处理平均耗时
- 向量库块总数
- 知识库文档总数

告警阈值：
- 失败率 > 5%
- 处理队列积压 > 100
- 单任务处理时间 > 5分钟
```

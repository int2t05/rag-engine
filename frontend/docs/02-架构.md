# 前端架构

## 1. 技术栈

| 层级 | 选型 |
|------|------|
| 框架 | Next.js 14（App Router） |
| 语言 | TypeScript |
| 样式 | Tailwind CSS 3.x + 全局 [`globals.css`](../src/app/globals.css) |
| 渲染 | Dashboard 业务页多为 `"use client"` + 浏览器端数据获取 |

## 2. 目录结构（逻辑分层）

```
frontend/src/
├── app/                    # App Router：页面、布局、元数据
│   ├── layout.tsx          # 根布局：字体、全局 body
│   ├── login|register/     # 认证页
│   └── dashboard/          # 需登录：layout 做 token 门禁
├── components/             # 可复用 UI（含 chat、Toast、ConfirmDialog 等）
├── hooks/                  # 页面级数据与副作用（会话、评估详情、文档流水线）
└── lib/                    # 无 UI 的工具与 API
    ├── api/                # HTTP 客户端 + DTO + 分模块 endpoints
    ├── routes.ts           # PATH 与面包屑
    ├── api-errors.ts       # FastAPI detail 解析
    ├── chat-stream.ts      # SSE 行与历史消息中的 RAG 上下文解析
    ├── evaluation-task-utils.ts
    └── …
```

**分层关系**（与重构计划一致）：

```text
页面 (app/.../page.tsx)
    → Hooks (hooks/useXxx.ts)  [可选：复杂页]
    → API (lib/api/endpoints.ts → client.fetchApi)
    → 后端 FastAPI
```

## 3. API 层设计

| 文件 | 职责 |
|------|------|
| [`api/types.ts`](../src/lib/api/types.ts) | 与后端 Pydantic 对齐的 TypeScript 接口 |
| [`api/client.ts`](../src/lib/api/client.ts) | `API_BASE`、`DEFAULT_TIMEOUT_MS`、`ApiError`、`fetchApi`、`api.get/post/put/delete` |
| [`api/endpoints.ts`](../src/lib/api/endpoints.ts) | `authApi`、`knowledgeBaseApi`、`chatApi`、`llmConfigApi`、`evaluationApi` |
| [`api/index.ts`](../src/lib/api/index.ts) | Barrel 导出，业务代码统一 `import … from "@/lib/api"` |

**特殊点**：

- **`chatApi.sendMessage`**：使用原生 `fetch` 直连接口，不走 `fetchApi` 的 Abort 超时封装，以便长连接 SSE；鉴权头与 `API_BASE` 与客户端一致。
- **204**：`fetchApi` 对 DELETE 无正文等情况返回 `null` 类型占位。

## 4. 错误与提示

- 业务错误：后端 JSON `detail` 经 [`parseFastApiErrorBody`](../src/lib/api-errors.ts) 转为可读字符串；抛出 [`ApiError`](../src/lib/api/client.ts)（含 `status`）。
- 全局轻提示：[`Toast`](../src/components/Toast.tsx)；危险操作：[`ConfirmDialog`](../src/components/ConfirmDialog.tsx)。

## 5. 路由与布局

- [`PATH`](../src/lib/routes.ts)：所有 `Link` / `router.replace` 的单一来源。
- **面包屑**：`breadcrumbsForPath(pathname)` 与侧边导航同一套路径认知。
- **全宽内容区**：`DASHBOARD_FULL_BLEED_PATHS`（当前含对话页）使 `main` 去 padding，便于聊天占满可视区域。

## 6. 与「业务模块」的映射（概要）

| 后端前缀（概念上） | 前端聚合 |
|--------------------|----------|
| `/api/auth` | `authApi` + 登录/注册页 |
| `/api/knowledge-base` | `knowledgeBaseApi` + `useDocumentPipeline`（知识库详情「新文件」上传）；文档详情页「替换文件」用 `replaceDocument(…, chunkParams)`（Query 带分块参数） |
| `/api/chat` | `chatApi` + `useChatSession` |
| `/api/evaluation` | `evaluationApi` + `useEvaluationTaskDetail` + `evaluation-task-utils` |
| `/api/llm-configs` | `llmConfigApi` + 模型配置页 |

详细操作顺序见 [03-业务逻辑.md](./03-业务逻辑.md)。

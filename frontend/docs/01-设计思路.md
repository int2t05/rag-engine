# 前端设计思路

## 1. 目标与边界（MVP）

- **目标**：为后端已提供的认证、知识库、对话（RAG 流式）、评估、LLM 配置能力提供**可用、可维护**的管理界面，优先保证主路径稳定，而非功能堆砌。
- **边界**：
  - 不引入全站 OpenAPI codegen、不默认接入 React Query 等重型数据层（降低配置与心智负担）。
  - 不在前端伪造后端不存在的接口或数据结构。
  - 流式对话保持独立 `fetch`，避免被通用 JSON 请求的默认超时绑死。

## 2. 与后端的契约关系

- 前端将 FastAPI 暴露的 REST（及 SSE）视为**唯一数据源**；DTO 类型定义在 `[src/lib/api/types.ts](../src/lib/api/types.ts)`，与 `backend/app/schemas` 及路由响应对齐。
- **消息历史**：后端无单独的「消息列表」GET；历史来自 `**GET /api/chat/{id}`** 返回体中的 `messages` 字段。界面切换会话时通过 `chatApi.get` 拉全量再渲染。
- **环境变量**：
  - `NEXT_PUBLIC_API_URL`：后端根地址（默认 `http://localhost:8000`）。
  - `NEXT_PUBLIC_API_TIMEOUT_MS`：非流式请求超时（毫秒，有合法范围限制），详见 `[client.ts](../src/lib/api/client.ts)`。

## 3. 体验与视觉取向（frontend-design 对齐）

- **字体**：根布局使用 **Fraunces（展示）+ IBM Plex Sans（正文）**（`[src/app/layout.tsx](../src/app/layout.tsx)`），避免默认 Inter/Roboto 等「通用 AI 界面」观感。
- **色彩与 token**：在 `[tailwind.config.ts](../tailwind.config.ts)` 中维护 `ink`、`canvas`、`surface`、`accent` 等语义色；业务组件**一律优先使用 token**，主按钮与链接使用 `accent`，不在页面中堆叠 `gray-*` / `blue-*`。完整对照、按钮与禁用态约定见 **[04-设计系统.md](./04-设计系统.md)**。
- **状态语义**：评估任务等使用 `[EvaluationStatusBadge](../src/components/EvaluationStatusBadge.tsx)`（琥珀 / 天青 / 翠 / 玫），与「待执行 / 执行中 / 已完成 / 失败」对应，便于扫读；此类状态色**不**与品牌 `accent` 混用。
- **背景与动效**：`[globals.css](../src/app/globals.css)` 使用低对比多层径向渐变作为页面底色；`.stagger-reveal` 为列表子项提供错开入场，避免单调闪现。
- **对话与知识库**：对话页、知识库详情、评估新建等已与 Dashboard 壳层统一为同一套 token（含 RAG 配置卡片、气泡、Toast、确认框）。

## 4. 鉴权与路由保护

- **令牌**：JWT 存 `localStorage`（键名 `token`）；请求头 `Authorization: Bearer <token>` 由 `[fetchApi](../src/lib/api/client.ts)` 统一附加。
- **保护方式**：`[dashboard/layout.tsx](../src/app/dashboard/layout.tsx)` 在客户端检测无 token 时 `replace` 至登录页；未使用 Edge Middleware 读 Cookie（与当前存 token 方式一致）。
- **401**：`fetchApi` 内清除 token 并跳转 `/login`，避免半登录态反复报错。

## 5. 可维护性约定

- **路由常量**：一律从 `[src/lib/routes.ts](../src/lib/routes.ts)` 的 `PATH` 引用，避免魔法路径字符串分叉。
- **副作用与复杂状态**：集中在 `src/hooks`（如 `useChatSession`、`useEvaluationTaskDetail`、`useDocumentPipeline`），页面以布局与组合为主。
- **注释**：新模块顶部使用中文 JSDoc 说明依赖的后端路径、轮询/SSE 行为及边界情况，便于后人对接口变更做 diff。
